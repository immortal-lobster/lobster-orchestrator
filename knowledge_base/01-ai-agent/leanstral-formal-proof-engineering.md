## 🚀 产品概述

### 核心定位
```
Leanstral = Lean + Astral
  - Lean: 定理证明器 (形式化验证)
  - Astral: Mistral AI 的编码助手品牌

使命：用形式化证明确保代码正确性
愿景：让每一行代码都可验证、可证明、可信赖
```

### 关键特性
```
✅ AI 生成代码 + 形式化证明
✅ 自动验证代码正确性
✅ 数学级严谨性保证
✅ 开源免费使用
✅ 集成主流 IDE (VSCode, JetBrains)
```

## 🔬 技术架构

### 核心组件
```
┌─────────────────────────────────────┐
│  Leanstral AI Engine                │
│  - 代码生成 (Mistral Large 2)       │
│  - 证明生成 (Lean 4 定理证明器)      │
│  - 验证循环 (AI ↔ 证明器)           │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  Formal Verification Layer          │
│  - 类型检查                         │
│  - 定理证明                         │
│  - 反例生成                         │
└─────────────────────────────────────┘
              ↓
┌─────────────────────────────────────┐
│  IDE Integration                    │
│  - VSCode 扩展                      │
│  - JetBrains 插件                   │
│  - 实时验证反馈                     │
└─────────────────────────────────────┘
```

### 工作流程
```
1. 用户输入需求 (自然语言或类型签名)
2. AI 生成候选代码
3. 形式化验证器检查正确性
4. 如果验证失败:
   - 生成反例
   - AI 根据反例修正代码
   - 返回步骤 3
5. 验证成功 → 输出代码 + 证明
```

## 🛠️ 技术实现细节

### Lean 4 集成
```lean
-- 示例：验证排序算法正确性
theorem sort_correct (xs : List Nat) :
  IsSorted (leanstral_sort xs) ∧
  IsPermutation (leanstral_sort xs) xs := by
  leanstral_auto  -- AI 自动生成证明
```

### AI 证明生成
```
输入：类型签名 + 规范
  def safe_divide (a b : Int) : Option Int
  requires: b ≠ 0
  ensures: result.isSome → result.get * b = a

AI 输出:
  - 实现代码
  - 形式化证明
  - 边界条件处理
```

### 验证循环
```
迭代 1: AI 生成代码 → 验证失败 (除零风险)
迭代 2: AI 添加检查 → 验证失败 (溢出风险)
迭代 3: AI 使用大整数 → 验证成功 ✓

总迭代次数：3 次
总时间：<10 秒
```

## ⚠️ 挑战与局限

### 技术挑战
```
1. 证明生成复杂度
   - 某些定理需要人工干预
   - AI 可能陷入无限循环
   - 需要智能超时机制

2. 性能开销
   - 形式化验证增加编译时间
   - 大型项目验证可能需要小时级
   - 需要增量验证优化

3. 学习曲线
   - 开发者需要理解类型理论
   - 规范编写需要训练
   - 需要更好的 UX/UI
```

### 采用障碍
```
1. 文化阻力
   - "测试就够了"心态
   - 形式化方法被视为"过度工程"
   - 需要教育市场

2. 集成成本
   - 现有代码库迁移成本高
   - 需要重构代码
   - 团队培训时间

3. 工具生态
   - 第三方库缺乏形式化规范
   - 需要社区共建
   - 鸡生蛋问题
```
