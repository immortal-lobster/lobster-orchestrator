## 🎯 核心问题

**地图 API 的时效性缺陷**:
- Google Maps 能告诉你"餐厅 4.2 星，营业到 10 点"
- 但**无法**告诉你：厨师上个月离职、等待时间翻倍、本地人已转向别处
- 现有地图 API 仅提供**固定快照**，无实时上下文

**市场规模**:
- ~40% 的 Google 搜索涉及本地上下文
- 高达 20% 的 LLM prompt 涉及本地信息
- 25-30% 的场所每年 churn (关闭/搬迁/更名)
- 世界不发射结构化"我关闭了"事件 - 需主动检测

**Agent 时代的放大**:
- Agent 开始在现实世界搜索/预订/购物
- 问题规模扩大 10 倍
- 基准测试：即使是最好的 LLM，也有 1/12 的本地查询出错

## 🚀 Voygr 是什么

**定义**: 无限的、可查询的场所画像 (infinite, queryable place profile)

**核心价值**: 结合准确的场所数据 + 新鲜的网络上下文 (新闻/文章/事件)

**创始人背景**:
- **Vlad**: Google Maps APIs 工作经验 + 共乘/旅行行业
- **Yarik**: Apple/Google/Meta ML/Search 基础设施负责人 (产品日活数亿用户)

**洞察**: 没人把"场所数据新鲜度"当作基础设施来建设。

## 🔧 核心产品

### 1. Business Validation API

**功能**: 判断企业是否**实际运营**

**检测能力**:
- ✅ 正常营业
- ❌ 已关闭
- 🔄 已更名/品牌重塑
- ⚠️ 数据无效/冲突

**技术方法**:
- 聚合多数据源
- 检测冲突信号
- 返回结构化 verdict

**定位**: "持续集成，但针对物理世界"

**GitHub**: [voygr-tech/dev-tools](https://github.com/voygr-tech/dev-tools)

### 2. Place Intelligence API (规划中)

**功能**: 无限可查询的场所画像

**数据维度**:
- 基础信息：名称/地址/营业时间/评分
- 实时上下文：新闻提及/社交媒体讨论/事件影响
- 趋势分析：人流变化/评价趋势/竞争动态
- 风险信号：关闭风险/服务质量下降/管理层变动

**查询示例**:
```
"找出过去 3 个月内厨师更换的意大利餐厅"
"显示受新地铁线路影响的商业区"
"识别评分下降但实际质量未变的场所"
```

## 🤖 Agent 时代的机会

### 1. Agent 本地查询基础设施
**问题**: Agent 帮用户预订餐厅/购买商品/预约服务，但数据过时导致错误决策。

**Voygr 价值**:
- 实时验证场所营业状态
- 提供最新服务质量信号
- 减少 Agent 幻觉 (基于事实数据)

**用例**:
```python
# Agent 查询示例
response = voygr.validate_business(
    place_id="ChIJN1t_tDeuEmsRUsoyG83frY4",
    check_fields=["operating_status", "recent_reviews", "news_mentions"]
)

if response.operating_status == "CLOSED_PERMANENTLY":
    # Agent 自动寻找替代场所
    alternatives = voygr.search_nearby(
        location=response.location,
        category=response.category,
        min_rating=4.0
    )
```

### 2. 多 Agent 协作中的场所协调
**场景**: 多个 Agent 协作规划活动 (会议/聚餐/旅行)。

**Voygr 作用**:
- 统一场所数据源 (避免 Agent 之间信息冲突)
- 实时同步场所状态变化
- 提供决策依据 (评分/人流/事件)

### 3. 本地化 RAG 增强
**问题**: LLM 本地知识容易过时 (训练数据截止)。

**Voygr 方案**:
- RAG 检索实时场所数据
- 注入 LLM 上下文窗口
- 生成准确的本地建议

## ⚠️ 风险与挑战

### 技术风险
```
1. 数据源依赖
   - 依赖第三方数据 (新闻/社交媒体/政府记录)
   - 数据源 API 变更/关闭风险
   - 对策：多元化数据源 + 自建爬虫

2. 检测准确性
   - 误报 (营业中判为关闭) 损害商家利益
   - 漏报 (已关闭判为营业) 损害用户体验
   - 对策：人工审核 + 用户反馈循环

3. 规模扩展
   - 全球数亿场所，全量监控成本高
   - 对策：优先高价值场所 + 按需检测
```

### 市场风险
```
1. 巨头竞争
   - Google Maps 可能推出类似功能
   - 对策：专注实时性 + 开放 API + 开发者生态

2. 采用门槛
   - 开发者需改变现有工作流
   - 对策：SDK + 示例代码 + 免费额度

3. 数据隐私
   - 场所数据可能涉及隐私 (如家庭企业)
   - 对策：合规审查 + 隐私保护设计
```

## 📈 行业趋势

### 1. 从"静态地图"到"动态世界模型"
- **过去**: 地图 = 固定地理数据
- **现在**: 地图 = 实时世界状态 (场所/事件/人流)
- **未来**: 地图 = 预测性世界模型 (趋势/风险/机会)

### 2. 从"人类查询"到"Agent 查询"
- **过去**: API 设计面向人类 (阅读友好)
- **现在**: API 设计面向 Agent (机器可读/结构化)
- **差异**: 
  - 人类：容忍模糊/不完整
  - Agent: 需要精确/完整/可验证

### 3. 从"单一数据源"到"多源融合"
- **过去**: Google Maps 单一权威源
- **现在**: 多源聚合 + 冲突检测 + 置信度评分
- **技术**: 数据融合/冲突解决/时间序列分析
