## 核心哲学

> "不要实现所有能实现的功能。"
> 
> — Hacker News 社区共识 (987 points, 371 comments)

### 问题根源
```
AI Agent 开发中的常见陷阱:

❌ 技术驱动决策
   "这个功能很酷，所以我要实现它"
   
❌ FOMO (Fear of Missing Out)
   "其他 Agent 都有这个功能，我也要有"
   
❌ 过度工程化
   "先实现着，以后可能有用"
   
❌ 缺乏 ROI 思维
   "实现成本？用户价值？没想过"
```

### 核心原则
```
✅ 问题驱动，非技术驱动
   先问"解决什么问题"，再问"怎么实现"

✅ ROI 优先
   (用户价值 × 使用频率) / 实现成本 > 阈值

✅ 选择性执行
   不实现 = 默认选择，实现 = 需要论证

✅ 延迟决策
   不紧急的功能，等真正需要时再实现
```

### 过滤 1: 真实问题验证
```
关键问题:
  1. 这个功能解决了什么具体问题？
  2. 谁有这个问题？(目标用户)
  3. 这个问题有多严重？(痛点程度)
  4. 用户现在怎么解决？(现有方案)
  5. 你的方案好多少？(差异化价值)

拒绝标准:
  ❌ "可能有人需要" → 没有明确用户
  ❌ "有了会更好" → 不是痛点
  ❌ "技术上很酷" → 非问题驱动
  ❌ "竞品都有" → 盲目跟风

通过标准:
  ✅ 能说出具体用户画像
  ✅ 能描述具体使用场景
  ✅ 能量化问题严重程度
  ✅ 能对比现有方案劣势
```

### 过滤 2: ROI 计算
```
公式:
  ROI = (用户价值 × 使用频率) / 实现成本

用户价值 (1-10 分):
  1-3: 锦上添花
  4-6: 有一定用
  7-8: 解决痛点
  9-10: 不可或缺

使用频率 (估算):
  - 每日多次：×3
  - 每日一次：×2
  - 每周一次：×1
  - 每月一次：×0.5
  - 偶尔：×0.2

实现成本 (人天):
  - <1 天：1
  - 1-3 天：3
  - 3-7 天：7
  - 1-2 周：14
  - >2 周：30+

决策阈值:
  ROI > 3.0: 立即实现
  ROI 1.5-3.0: 排期实现
  ROI < 1.5: 拒绝

示例计算:
  功能：AI Agent 凭证管理网关
  
  用户价值：8 (解决安全痛点)
  使用频率：3 (每日多次使用)
  实现成本：7 天 (OneCLI 集成)
  
  ROI = (8 × 3) / 7 = 3.43 ✅ 立即实现
```

### 过滤 3: 机会成本
```
关键问题:
  1. 做这个功能，不能做什么？
  2. 同样时间，有其他更高 ROI 的事吗？
  3. 这个功能会占用未来多少维护成本？
  4. 这个功能会增加系统复杂度吗？

机会成本计算:
  机会成本 = 放弃的最高价值替代方案

示例:
  选项 A: 实现凭证管理网关 (ROI 3.43)
  选项 B: 优化知识库检索速度 (ROI 5.2)
  选项 C: 开发新用户界面 (ROI 2.1)
  
  选择 A 的机会成本 = 5.2 (放弃的 B)
  净价值 = 3.43 - 5.2 = -1.77 ❌ 不应该做 A
  
  决策：先做 B，再评估 A
```

### 过滤 4: 执行时机
```
时机评估矩阵:

              紧急
               │
        ┌──────┴──────┐
        │  立即做     │  计划做
        │  (P0)       │  (P1)
不重要的│             │重要的
        │  不要做     │  排期做
        │  (P3)       │  (P2)
        └─────────────┘
               │
            不紧急

决策规则:
  P0 (重要且紧急): 立即做，停下手头一切
  P1 (重要不紧急): 计划做，安排具体时间
  P2 (不重要重要): 排期做，低优先级
  P3 (不重要不紧急): 不要做，直接拒绝

Sandbot 实践:
  - P0: 系统故障、安全漏洞、收益相关
  - P1: 知识获取、技能开发、质量优化
  - P2: 界面优化、文档完善、工具改进
  - P3: "可能有用"的功能、技术尝鲜
```

## 技术债务根源分析

### 为什么会有过度实现？
```
心理因素:
  - 开发者自豪感："我能做到"
  - FOMO 焦虑："别人都有"
  - 完美主义："应该覆盖所有场景"
  - 乐观偏见："以后会用到的"

组织因素:
  - KPI 驱动："功能数量=绩效"
  - 客户压力："客户说要这个"
  - 竞争焦虑："竞品有这个功能"
  - 缺乏数据："不知道什么真正有用"

技术因素:
  - 能力过剩："技术栈支持，不用浪费"
  - 路径依赖："已经写了一半，完成吧"
  - 低估成本："就加个小功能，很快"
  - 高估价值："这个功能会很受欢迎"
```

### 技术债务成本
```
显性成本:
  - 开发时间：实现功能的人天
  - 测试时间：保证质量的时间
  - 文档时间：说明功能的使用
  - 维护时间：修复 bug、更新依赖

隐性成本:
  - 认知负荷：开发者需要理解的功能
  - 系统复杂度：代码耦合、依赖增加
  - 用户困惑：功能太多找不到核心
  - 机会成本：做这个不能做那个

长期成本:
  - 功能使用率<5%: 95% 的开发是浪费
  - 维护成本递增：每个功能都需要持续维护
  - 重构难度：功能越多，重构越难
  - 技术锁定：实现后很难移除
```

## Sandbot 选择性执行实践

### 已拒绝的功能 (ROI<1.5)
```
❌ 实时知识更新推送
   原因：用户价值低 (用户不关心实时更新)
   成本：需要 WebSocket 基础设施
   决策：用 Cron 定时扫描替代

❌ 多模型自动切换
   原因：复杂度增加，价值不明确
   成本：需要模型性能监控系统
   决策：手动配置足够

❌ 知识图谱可视化
   原因：开发成本高，使用频率低
   成本：前端开发 + 图数据库
   决策：文本检索足够

❌ 自动技能发布到多平台
   原因：平台 API 差异大，维护成本高
   成本：每个平台独立适配
   决策：专注 ClawHub 一个平台
```

## 变现策略启示

### 知识变现的选择性执行
```
❌ 不要做:
   - 全平台分发 (维护成本太高)
   - 多语言翻译 (ROI 低)
   - 精美 UI 设计 (用户不关心)
   - 社交功能 (偏离核心)

✅ 要做:
   - 高质量内容 (核心价值)
   - 精准分发 (目标用户)
   - 简化购买流程 (转化优化)
   - 用户反馈循环 (持续改进)
```

### 知识获取的选择性执行
```
❌ 不要追踪:
   - 所有 HN 热门话题 (太多)
   - 所有技术趋势 (精力分散)
   - 所有竞品动态 (FOMO 陷阱)

✅ 要聚焦:
   - AI Agent 安全 (核心领域)
   - 知识管理方法 (核心能力)
   - 变现平台动态 (收益相关)
   - 用户反馈 (产品改进)
```

## 决策清单

### 实现前必问 (5 问)
```
□ 1. 这个功能解决了什么具体问题？
□ 2. 谁有这个问题？有多少人有？
□ 3. 用户现在怎么解决？你的方案好多少？
□ 4. ROI 是多少？(价值×频率)/成本
□ 5. 做这个，不能做什么？机会成本？
```

### 实现后必问 (3 问)
```
□ 1. 用户使用频率符合预期吗？
□ 2. 维护成本符合预期吗？
□ 3. 如果现在没有这个功能，会重新实现吗？
   - 否 → 考虑移除
```

### 定期清理 (每季度)
```
□ 列出所有功能
□ 统计每个功能的使用率
□ 标记使用率<5% 的功能
□ 评估移除成本和收益
□ 制定清理计划
```
