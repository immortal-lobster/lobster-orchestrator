## 核心工作流

### 1. 探索阶段 (Exploration)
```
目标：快速了解问题空间和技术选项

LLM 用途:
  - 解释陌生概念
  - 提供技术对比分析
  - 生成学习路径

提示词示例:
  "我需要实现 X 功能，有哪些技术方案？
   请对比 A/B/C 三种方案的优缺点。"
```

### 2. 脚手架阶段 (Scaffolding)
```
目标：建立项目结构和基础代码

LLM 用途:
  - 生成项目模板
  - 创建配置文件
  - 设置测试框架

提示词示例:
  "创建一个 Python 项目，包含:
   - pytest 测试
   - black/ruff 格式化
   - GitHub Actions CI
   - 文档骨架"
```

### 3. 迭代开发阶段 (Iteration)
```
目标：逐步实现功能

LLM 用途:
  - 生成函数/类实现
  - 代码审查和优化建议
  - 调试帮助

关键技巧:
  - 小步迭代 (每次 50-200 行)
  - 立即测试生成的代码
  - 保留人工审查环节

提示词示例:
  "实现一个函数，输入 X，输出 Y。
   要求：处理边界情况 Z，时间复杂度 O(n)。"
```

### 4. 重构阶段 (Refactoring)
```
目标：改善代码质量

LLM 用途:
  - 识别代码异味
  - 提供重构建议
  - 自动化重构

提示词示例:
  "审查这段代码，指出:
   1. 潜在 bug
   2. 性能问题
   3. 可读性改进点
   4. 测试覆盖率缺口"
```

## 关键原则

### ✅ 应该做的
1. **LLM 是副驾驶，不是自动驾驶**
   - 始终理解生成的代码
   - 对关键逻辑进行人工验证
   - 保持架构决策的主导权

2. **小步快跑**
   - 每次请求聚焦单一任务
   - 生成后立即测试
   - 累积小胜利而非大跳跃

3. **上下文管理**
   - 提供足够的背景信息
   - 分享相关代码片段
   - 明确约束条件和边界

4. **迭代反馈**
   - 告诉 LLM 什么有效/无效
   - 基于错误信息调整提示
   - 建立对话式开发节奏

### ❌ 避免的陷阱
1. **盲目信任**
   - LLM 会编造 API、库、行为
   - 始终验证生成的代码
   - 不要复制粘贴不理解的代码

2. **过大请求**
   - "帮我写整个应用" → 失败率高
   - 分解为小任务 → 成功率高

3. **忽视测试**
   - LLM 生成的代码需要测试
   - 测试驱动开发仍然适用
   - 自动化测试是安全网

4. **上下文污染**
   - 过长的对话历史会降低质量
   - 定期重启对话或总结上下文
   - 关键决策点开启新对话

## 高级技巧

### 1. 提示词模板化
```markdown
# 角色设定
你是一位资深 [语言] 工程师，擅长 [领域]。

# 任务描述
我需要 [具体任务]。

# 上下文
- 项目类型：[类型]
- 现有代码：[粘贴相关代码]
- 约束条件：[性能/安全/兼容性要求]

# 输出要求
- 代码风格：[规范]
- 测试：[需要/不需要]
- 文档：[内联注释/API 文档]
- 解释：[详细/简洁/无]
```

### 2. 错误驱动开发
```
1. 让 LLM 生成初始实现
2. 运行测试，捕获错误
3. 将错误信息反馈给 LLM
4. 迭代直到通过

优势:
  - 快速收敛到正确实现
  - 学习错误模式和修复方法
  - 建立测试优先的习惯
```

### 3. 代码审查工作流
```
第一轮：功能正确性
  "这段代码能正确实现 X 功能吗？有什么边界情况没处理？"

第二轮：代码质量
  "从可读性、可维护性角度，有哪些改进建议？"

第三轮：性能优化
  "这段代码有性能瓶颈吗？如何优化？"

第四轮：安全审计
  "有什么安全隐患？如何修复？"
```

## 风险与缓解

### 风险 1: 代码同质化
```
问题：过度依赖 LLM 导致代码风格单一
缓解：
  - 保持人工架构设计
  - 定期代码审查
  - 鼓励团队多样性
```

### 风险 2: 技能退化
```
问题：年轻开发者过度依赖，基础能力下降
缓解：
  - 强制理解生成的代码
  - 定期无 AI 编程练习
  - 重视代码审查和解释
```

### 风险 3: 安全漏洞
```
问题：LLM 可能生成有漏洞的代码
缓解：
  - 安全扫描工具 (SAST/DAST)
  - 依赖项审计
  - 安全编码培训
```
