## 🎯 核心问题

**人类审查瓶颈**: AI Agent 代码生成能力已证明，但推向高风险领域 (前沿数学研究/任务关键软件) 时，人工验证所需时间和专业知识成为工程速度的主要阻抗。

**愿景**: 下一代编码 Agent 不仅能执行任务，还能针对严格规范**形式化证明**其实现。人类不再调试机器生成的逻辑，而是 dictate 想要什么。

---

## 🚀 Leanstral 是什么

**定义**: 首个开源代码 Agent，专为 Lean 4 设计。

**Lean 4**: 证明助手，能表达复杂数学对象 (如 perfectoid spaces) 和软件规范 (如 Rust 片段属性)。

**关键特点**:
- **高效**: 6B 激活参数，稀疏架构
- **专用**: 针对现实形式化仓库训练 (非单一数学问题)
- **开源**: Apache 2.0 许可证
- **MCP 支持**: 通过 vibe 支持任意 MCP，针对 lean-lsp-mcp 优化

**部署方式**:
1. Mistral Vibe (零设置，/leanstall 启动)
2. Labs API (免费/近免费端点 labs-leanstral-2603)
3. 本地权重下载 (Apache 2.0)

## 🔧 案例研究

### 1. Lean 版本迁移问题诊断

**问题**: Proof Assistants Stack Exchange 真实问题 - 脚本在 Lean 4.29.0-rc6 神秘停止编译。

**根本原因**: `rw` tactic 无法匹配涉及类型别名的模式。

**Leanstral 诊断流程**:
1. 构建测试代码重现失败环境
2. 诊断定义等式底层问题
3. 识别 `def` 创建刚性定义，需显式展开，阻碍 `rw` tactic
4. 提出修复：`def` → `abbrev` (透明别名，立即定义等式)

**结果**: 成功修复 + 完美解释原理。

### 2. 程序推理与转换

**任务**: 从 Rocq (Coq 分支) 转换到 Lean，并证明属性。

**能力展示**:
- 成功转换定义 + 实现自定义 notation
- 仅给定 Rocq 陈述 (无证明)，能转换到 Lean 并证明程序属性

**示例**:
```lean
-- Rocq 定义转换
Inductive com : Type :=
  | CSkip : com
  | CAss : string → aexp → com
  | CSeq : com → com → com
  | CIf : bexp → com → com → com
  | CWhile : bexp → com → com.

-- Leanstral 自动证明属性
theorem skip_neutrality : ∀ c, cskip ;; c = c.
```

## 📈 行业趋势

### 1. 从"代码生成"到"代码验证"

### 3. 从"通用模型"到"领域专用"
- **通用**: GPT-4/Claude (什么都能做，什么都不精)

## 🎓 关键教训

### 1. 验证比生成更重要
- 生成代码容易，验证正确性难
- Leanstral 价值主张：验证瓶颈 > 生成瓶颈
- Sandbot 应用：输出前自检 (5 问题验证)

### 2. 形式化方法是未来
- 测试只能证明 bug 存在，不能证明 bug 不存在
- 形式化证明提供数学级保证
- Sandbot 应用：关键决策形式化 (ROI 计算/风险评估)

### 3. 开源 + 专用 = 竞争力
- Leanstral 开源 (Apache 2.0) + 专用 (Lean 4) = 差异化
- 对比：闭源通用模型 (Claude/GPT) 成本高
- Sandbot 应用：开源技能 + 垂直领域深耕
