## 核心洞察

### 编程范式的转变
```
传统编程：
  - 需要手动编写每一行代码
  - 缺陷率随代码量增长
  - 2-3 天后代码变得难以维护

LLM 辅助编程 (Codex 5.2 / Opus 4.6 之后):
  - 缺陷率显著低于手写代码
  - 数万行代码仍可维护
  - 工程技能从"写代码"转向"架构设计"
```

### 关键发现
1. **不同人使用 LLM 效果差异巨大** - 工作流细节决定成败
2. **工程技能没有消失，只是转移了** - 架构能力 > 编码能力
3. **技术熟悉度仍然重要** - 不熟悉的技术栈仍会产生混乱代码
4. **模型能力在进化** - 从检查每行 → 检查函数 → 检查架构

## 核心工作流

### 阶段 1: 探索 (Exploration)
```
目标：理解问题空间和技术选项

活动:
  - 用 LLM 调研技术方案
  - 生成多个设计选项
  - 评估权衡 (性能/复杂度/可维护性)

LLM 提示示例:
  "我需要实现 X 功能，有哪些技术方案？
   请列出 3-5 种方案，每种方案的：
   - 优点/缺点
   - 适用场景
   - 实现复杂度
   - 长期维护成本"

产出:
  - 技术方案对比文档
  - 架构决策记录 (ADR)
```

### 阶段 2: 脚手架 (Scaffolding)
```
目标：建立项目结构和基础代码

活动:
  - 生成项目骨架
  - 定义核心接口
  - 设置测试框架
  - 配置 CI/CD

LLM 提示示例:
  "基于我们选择的技术方案，请生成：
   1. 项目目录结构
   2. 核心模块的接口定义
   3. 单元测试框架配置
   4. 基础 CI/CD 配置"

关键原则:
  - 先定义接口，再实现细节
  - 测试驱动开发 (TDD)
  - 文档即代码

产出:
  - 可运行的项目骨架
  - 核心接口定义
  - 测试基础设施
```

### 阶段 3: 迭代开发 (Iterative Development)
```
目标：逐步实现功能

活动:
  - 小步提交 (每次一个功能点)
  - 持续测试验证
  - 代码审查 (人工 + 工具)

LLM 提示示例:
  "实现 X 功能，要求：
   1. 遵循已定义的接口
   2. 包含完整的单元测试
   3. 添加类型注解
   4. 编写文档字符串"

迭代循环:
  1. LLM 生成代码
  2. 运行测试
  3. 人工审查关键逻辑
  4. 修复问题
  5. 提交代码

关键原则:
  - 小步快跑 (每次提交 <200 行)
  - 测试先行
  - 人工审查架构级决策
```

### 阶段 4: 重构与优化 (Refactoring)
```
目标：提升代码质量

活动:
  - 识别代码异味
  - 重构复杂逻辑
  - 性能优化
  - 文档完善

LLM 提示示例:
  "审查这段代码，识别：
   1. 代码异味 (code smells)
   2. 复杂度高的函数
   3. 重复代码
   4. 潜在的性能问题
   
   然后提供重构建议"

工具链:
  - 静态分析 (linters)
  - 复杂度分析
  - 性能分析器
  - 文档生成器
```

## 关键原则

### 1. 架构优先 (Architecture First)
```
✅ 正确做法:
  - 先设计整体架构
  - 定义清晰的模块边界
  - 确定数据流和控制流
  - 文档化架构决策

❌ 错误做法:
  - 直接让 LLM 生成代码
  - 边写边改架构
  - 忽视模块耦合
```

### 2. 测试驱动 (Test-Driven)
```
✅ 正确做法:
  - 先写测试用例
  - 让 LLM 实现通过测试的代码
  - 保持高测试覆盖率 (>80%)
  - 自动化测试执行

测试金字塔:
        /\
       /  \      E2E 测试 (10%)
      /----\
     /      \    集成测试 (20%)
    /--------\
   /          \  单元测试 (70%)
  /------------\
```

### 3. 渐进式验证 (Progressive Verification)
```
早期 LLM 时代:
  - 检查每一行代码
  - 验证每个函数
  - 完全人工审查

现代 LLM (2026):
  - 验证架构设计
  - 审查关键逻辑
  - 自动化测试覆盖
  - 工具辅助检查

未来趋势:
  - 可能只需验证高层目标
  - AI 自主完成实现和测试
```

### 4. 上下文管理 (Context Management)
```
策略:
  - 保持项目文档更新
  - 维护架构决策记录
  - 定期生成代码摘要
  - 使用 RAG 增强 LLM 上下文

工具:
  - 项目知识库
  - 代码索引 (如 Sourcegraph)
  - 文档自动生成
```

## 实战案例：Stavrobot 项目

### 项目概述
- **类型**: 个人 AI 助理
- **规模**: 数万行代码
- **开发周期**: 数周持续开发
- **维护状态**: 每日使用，持续迭代

### 核心功能
```
1. 日历管理
   - 智能可用性判断
   - 冲突检测与解决
   - 自动调度

2. 研究助理
   - 网络搜索
   - 信息整理
   - 报告生成

3. 自我扩展
   - 自主编写新代码
   - 功能迭代
   - Bug 修复

4. 任务管理
   - 提醒系统
   - 家务自动化
   - 优先级排序
```

### 架构设计
```
┌─────────────────────────────────────┐
│           用户接口层                 │
│  (Telegram / Web / CLI)            │
└─────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────┐
│           决策引擎                   │
│  (意图识别 / 任务规划 / 执行)       │
└─────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────┐
│           工具层                     │
│  (日历 API / 文件操作 / 网络搜索)    │
└─────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────┐
│           安全层                     │
│  (权限控制 / 操作审计 / 沙箱)       │
└─────────────────────────────────────┘
```

### 安全设计原则
```
1. 最小权限原则
   - 每个工具独立权限
   - 用户明确授权
   - 操作日志记录

2. 可解释性
   - 每个决策可追溯
   - 操作原因可查询
   - 审计日志完整

3. 故障安全
   - 默认拒绝未知操作
   - 异常立即停止
   - 用户确认关键操作
```

### 关键挑战与解决方案
```
挑战 1: 安全性 vs 可用性
  解决：分层权限模型 + 用户确认关键操作

挑战 2: 代码质量维护
  解决：严格测试 + 架构审查 + 小步迭代

挑战 3: 上下文管理
  解决：项目知识库 + RAG 增强
```

## 高级技巧

### 1. 提示工程 (Prompt Engineering)
```
好提示的要素:
  - 清晰的目标描述
  - 明确的约束条件
  - 示例输入/输出
  - 分步指导

示例模板:
  """
  任务：[清晰描述要完成的任务]
  
  约束:
  - [约束 1: 如性能要求]
  - [约束 2: 如兼容性要求]
  - [约束 3: 如代码风格]
  
  上下文:
  - [相关背景信息]
  - [已有代码/设计]
  
  输出格式:
  - [期望的输出结构]
  - [需要包含的内容]
  
  示例:
  [输入示例]
  [期望输出示例]
  """
```

### 2. 上下文窗口优化
```
策略:
  - 只包含必要上下文
  - 使用摘要替代全文
  - 分阶段处理大任务
  - 利用 RAG 检索相关信息

技巧:
  - 项目摘要文档 (<1000 字)
  - 模块接口定义
  - 关键决策记录
  - 测试用例摘要
```

### 3. 质量保障
```
自动化检查:
  - 代码格式化 (pre-commit)
  - 静态分析 (CI)
  - 单元测试 (每次提交)
  - 集成测试 (每日)

人工审查:
  - 架构变更
  - 安全相关代码
  - 性能关键路径
  - 公共 API 设计

度量指标:
  - 测试覆盖率 (>80%)
  - 代码复杂度 (<20)
  - 技术债务比率 (<5%)
  - Bug 逃逸率 (<1%)
```

## 常见陷阱与避免方法

### 陷阱 1: 过度依赖 LLM
```
症状:
  - 不理解生成的代码
  - 无法调试复杂问题
  - 架构决策完全交给 AI

避免方法:
  - 保持核心领域知识
  - 人工审查关键决策
  - 定期"无 AI"编程练习
```

### 陷阱 2: 测试不足
```
症状:
  - 测试覆盖率低
  - 回归 bug 频发
  - 重构困难

避免方法:
  - 测试驱动开发
  - CI 强制覆盖率
  - LLM 生成测试用例
```

### 陷阱 3: 架构腐化
```
症状:
  - 模块耦合度高
  - 技术债务累积
  - 新功能难以添加

避免方法:
  - 定期架构审查
  - 重构时间预算 (20%)
  - 架构决策记录
```

### 陷阱 4: 上下文丢失
```
症状:
  - LLM 忘记项目约定
  - 重复讨论已决策问题
  - 代码风格不一致

避免方法:
  - 维护项目知识库
  - 定期更新文档
  - 使用 RAG 检索
```
