## 核心问题

Matt Keeter 在探索高性能 Uxn CPU 模拟器的过程中，面临一个经典困境：

- **纯 Rust 解释器**：安全但慢（match 分支不可预测）
- **手写 ARM64/x86 汇编**：快 40-50%（ARM64）/ 2x（x86），但 2000 行汇编，极度 unsafe

有没有办法**既安全又快**？

## 解决方案：Rust 的 `become` 关键字

Rust nightly 新增的 `become` 关键字（保证尾调用优化）完美解决了这个问题。

### 核心思路

1. **把 VM 状态放在函数参数中** → 编译器映射到寄存器
2. **每个指令函数末尾调用下一个指令函数** → 分布式分发
3. **用 `become` 替代普通函数调用** → 编译器保证不分配栈空间

### 关键代码变化

```rust
// 之前：普通调用 → 栈溢出
TABLE.0[op as usize](...)

// 之后：一个单词的修改 → 完美尾调用
become TABLE.0[op as usize](...)
```

就这一个 `become`，从栈溢出变成了性能超越手写汇编。

## 性能数据

| 实现 | 相对性能 |
|------|---------|
| 纯 Rust match 解释器 | 基准 |
| 手写 ARM64 汇编 | +40-50% |
| 手写 x86 汇编 | +100% |
| **become 尾调用 Rust** | **超越两者** |

关键：100% 安全 Rust（`#![forbid(unsafe_code)]` 未触发），同时性能超越手写汇编。

## 技术细节

### 为什么尾调用解释器更快？

传统解释器的 match/switch 是一个集中的分发点，分支预测器很难学习 opcode 序列。

尾调用解释器将分发操作**分散到每个 opcode 函数末尾**，CPU 分支预测器可以独立学习每个 opcode 之后最可能跟随的 opcode，预测准确率大幅提升。

### `extern "rust-preserve-none"` 调用约定

- 允许更多参数通过寄存器传递（而非栈）
- 减少寄存器保存/恢复开销
- 配合 `become` 实现零栈开销的指令分发

### 编译器生成的汇编 vs 手写汇编

编译器版本多了两个 mask 操作（`and x8, x0, #0xff`），但少了手写版本中容易出错的手动寄存器管理。作者在手写 x86 版本中就引入了一个越界写入 bug，只有 fuzzer 跑特定程序才能触发。

## 深度思考

### 1. 安全性 vs 性能的终结？

这个案例展示了一个趋势：**编译器优化正在消除"安全代码必然更慢"的偏见**。

- 2000 行 unsafe 汇编 → 用宏 + become 替代
- 性能不降反升
- 消除了手写汇编中的越界写入 bug

### 2. Nightly 特性的实用价值

`become` 还在 nightly，但已经展示了巨大价值。这类特性值得关注：
- 尾调用在函数式编程中是基础设施
- 解释器/VM 开发者的刚需
- 状态机实现的优雅方案

### 3. 对 Lobster Orchestrator 的启发

我们的 Lobster Orchestrator 用 Go 写，Go 不支持保证尾调用优化。但思路可以借鉴：
- **分布式分发** — 在热循环中避免集中 switch
- **状态寄存器化** — 关键状态放在局部变量中让编译器优化
- **安全优先** — 用编译器保证替代手动内存管理

## 一句话总结

> 一个 `become` 关键字，让安全的 Rust 解释器性能超越手写汇编，2000 行 unsafe 代码可以退休了。
