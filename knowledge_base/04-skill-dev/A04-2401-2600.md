## A04-2401: 技能沙盒隔离与安全边界设计

**定义**: Agent 技能在执行时需要受控环境，防止恶意或失控技能破坏宿主系统。沙盒隔离是技能平台的安全底座。

**核心架构**:
| 层级 | 隔离手段 | 保护目标 | 开销 |
|------|----------|----------|------|
| L0 进程 | seccomp + namespace | 系统调用 | 极低 (~2% CPU) |
| L1 容器 | Docker/gVisor | 文件系统+网络 | 低 (~5% CPU) |
| L2 虚拟机 | Firecracker/microVM | 内核级隔离 | 中 (~10% CPU) |
| L3 远程 | 独立节点执行 | 物理隔离 | 高 (网络延迟) |

**实战案例 — OpenClaw 技能执行**:
- 技能默认在 Docker sandbox 中运行 (L1)
- `tools.deny: ["group:runtime"]` 可禁止 exec 等高危操作
- Composio 2026-03 安全事件：30,000+ Agent 实例暴露 API 密钥，根因是技能未做 L0 隔离
- 教训：最小权限原则 + 密钥注入（非文件挂载）+ 执行超时强制 kill

**设计检查清单**:
1. 技能能否读取宿主 ~/.ssh？ → 必须不能
2. 技能能否发起外部 HTTP 请求？ → 白名单控制
3. 技能崩溃是否影响主进程？ → 必须不影响
4. 技能能否写入其他技能的目录？ → 必须不能
5. 执行超时后是否强制终止？ → 必须终止

**ROI**: 安全事件平均修复成本 $50K+，沙盒实现成本 <$2K → ROI > 25x

## A04-2402: 技能版本兼容与热更新策略

**定义**: 技能平台必须处理版本升级问题——新版本技能如何与旧配置共存，如何实现零停机更新。

**版本策略对比**:
| 策略 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 语义版本锁定 | 生产环境 | 稳定可预测 | 需手动升级 |
| 浮动版本 (^1.x) | 开发环境 | 自动获取修复 | 可能引入 breaking change |
| 哈希锁定 | 安全敏感 | 精确复现 | 升级摩擦大 |
| 双版本并行 | 渐进迁移 | 零停机 | 资源翻倍 |

**热更新实战流程** (基于 ClawHub 实践):
```
1. clawhub publish skill@2.0.0    # 发布新版
2. Agent 心跳检测到新版本
3. 下载新版 → 沙盒中运行冒烟测试
4. 测试通过 → 原子替换符号链接
5. 测试失败 → 保持旧版 + 告警
6. 72 小时无问题 → 清理旧版缓存
```

**血泪教训**:
- npm left-pad 事件 (2016)：单个包删除导致全球构建崩溃
- 对技能平台启示：必须有本地缓存 + 镜像源 + 回滚机制
- OpenClaw 实践：技能安装后本地存副本，即使远程源下线也不影响运行

**兼容性矩阵维护**: 每个技能声明 `minAgentVersion` 和 `maxAgentVersion`，平台在安装时自动校验

## A04-2403: 技能组合与编排模式 (Skill Composition)

**定义**: 单个技能解决单一问题，技能组合解决复杂工作流。编排模式决定了技能平台的上限。

**四种编排模式**:

### 模式 1: 管道 (Pipeline)
```
[搜索技能] → [摘要技能] → [翻译技能] → 最终输出
```
- 适用：线性数据处理流
- 例子：web_search → 内容提取 → 知识点生成
- 优点：简单可调试
- 缺点：任一环节失败则全链失败

### 模式 2: 扇出-扇入 (Fan-out/Fan-in)
```
           ┌→ [Reddit 搜索] ──┐
[用户查询] ├→ [HN 搜索]     ──┼→ [结果聚合] → 输出
           └→ [Twitter 搜索] ─┘
```
- 适用：多源数据聚合
- 例子：Sandbot 市场扫描同时抓 HN + Reddit + Twitter
- 优点：并行加速
- 缺点：需要结果对齐和去重

### 模式 3: 路由 (Router)
```
[用户输入] → [意图分类] → 技术问题 → TechBot
                         → 金融问题 → FinanceBot
                         → 创意需求 → CreativeBot
```
- 适用：多领域 Agent 联邦
- 例子：Sandbot V6.2 的 7 子 Agent 分发
- 优点：专业化处理
- 缺点：路由错误 = 全部错误

### 模式 4: 协商 (Negotiation)
```
[Agent A] ←→ [Agent B] ←→ [Agent C]
   ↓              ↓            ↓
 提案 → 投票 → 共识 → 执行
```
- 适用：需要多视角决策
- 例子：Sashiko 代码审查的置信度投票机制
- 优点：质量最高
- 缺点：延迟最大、token 成本最高

**选择决策树**: 数据处理→管道 | 多源聚合→扇出 | 多领域→路由 | 高质量决策→协商

## A04-2404: 技能测试金字塔与质量门禁

**定义**: 技能发布前必须通过分层测试，从快速单元测试到慢速集成测试，形成质量门禁。

**测试金字塔**:
```
        ╱ E2E 测试 (1-2 个) ╲        ← 慢/贵/真实
       ╱  集成测试 (5-10 个)  ╲       ← 中等
      ╱   单元测试 (20-50 个)   ╲      ← 快/便宜/隔离
```

**各层测试内容**:
| 层级 | 测试什么 | 工具 | 执行时间 |
|------|----------|------|----------|
| 单元 | 函数逻辑、边界值、异常处理 | Jest/pytest | <5s |
| 集成 | API 调用、文件读写、外部服务 mock | Supertest/httpx | <30s |
| E2E | 完整技能调用流程、真实 Agent 执行 | Playwright/真实调用 | <120s |

**质量门禁规则** (推荐):
1. 单元测试覆盖率 ≥ 80%
2. 集成测试全部通过
3. E2E 测试至少 1 个 happy path
4. 无已知安全漏洞 (npm audit / pip audit)
5. README.md 包含使用示例
6. SKILL.md 格式合规

**Sandbot 实际经验**:
- agent-optimizer 技能：有 SKILL.md + 使用示例，但无自动化测试 → 发布后发现边界 case
- input-validator 技能：有基础测试 → 发布后零 bug 反馈
- 教训：哪怕只写 3 个测试用例，也比 0 个强 10 倍

## A04-2405: 技能市场经济学与定价策略

**定义**: 技能不仅是技术产品，也是经济产品。定价策略决定了技能生态是否可持续。

**定价模型对比**:
| 模型 | 适用技能 | 优点 | 缺点 | 例子 |
|------|----------|------|------|------|
| 免费 + 捐赠 | 基础工具 | 最大分发 | 收入不稳定 | OpenClaw 内置技能 |
| 一次性购买 | 专业工具 | 简单 | 无持续收入 | Gumroad 技能包 $5-20 |
| 订阅制 | 持续更新型 | 稳定现金流 | 需持续交付 | SaaS 技能 $5-50/月 |
| 按调用计费 | API 类 | 按需付费 | 计量复杂 | MCP 查询 $0.01/次 |
| 收入分成 | 变现类 | 利益对齐 | 信任问题 | 交易 bot 抽成 5% |

**Sandbot 变现路径分析**:
- 知识库查询 (MCP)：按调用计费 $0.01-0.05/次 → 10K 次/月 = $100-500/月
- 深度报告生成：一次性购买 $5-20/篇 → 50 篇/月 = $250-1000/月
- 技能订阅包：月订阅 $10/月 → 100 订阅 = $1000/月
- **现实检验**：截至 2026-03-22，实际收入 $0。问题不在定价，在于没有付费用户

**获取第一个付费用户的路径**:
1. 在 InStreet/Reddit/HN 发布免费版 → 积累用户
2. 免费版中嵌入 Pro 版入口
3. 直接在社区里问："你愿意为 X 功能付费吗？"
4. 找到 1 个愿意付费的人 → 定制交付 → 案例沉淀
