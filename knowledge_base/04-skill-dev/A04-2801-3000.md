### A04-2801: Skill Manifest 设计模式
- **定义**: Agent 技能的元数据声明文件（SKILL.md / manifest.json），描述技能的名称、触发条件、依赖、输入输出约定
- **核心原理**: 技能发现依赖结构化元数据匹配，而非全文搜索；description 字段是技能路由的第一判据
- **最佳实践**: description 用自然语言写"何时触发"而非"功能描述"；例如 "Use when user mentions sharing, permissions" 优于 "Permission management skill"
- **真实案例**: OpenClaw 的 `<available_skills>` 标签在系统提示词中列出所有技能 description，模型据此选择；匹配失败则跳过技能
- **关键参数**: name (唯一标识), description (触发条件), location (SKILL.md 路径), dependencies (运行时依赖)
- **陷阱**: description 写得太泛导致误触发，太窄导致漏触发；需要 A/B 测试不同措辞的命中率

### A04-2802: SKILL.md 内部结构规范
- **定义**: 技能指令文件的内部章节组织，指导 Agent 在加载技能后如何执行
- **核心原理**: Agent 读取 SKILL.md 后按顺序执行；文件本身就是 prompt，不是文档
- **推荐结构**: 1) 概述与触发条件 2) 前置检查（环境/权限） 3) 执行步骤（编号列表） 4) 输出格式 5) 错误处理 6) 示例
- **真实案例**: OpenClaw weather skill 的 SKILL.md 包含 wttr.in API 调用模板 + 格式化输出指令 + 降级方案（wttr.in 不可用时切 Open-Meteo）
- **关键参数**: 步骤数量建议 3-8 步；超过 10 步应拆分为子技能
- **陷阱**: 把 SKILL.md 写成 README（面向人类）而非指令（面向 Agent）；Agent 需要的是可执行步骤，不是背景知识

### A04-2803: 技能引用脚本与资产管理
- **定义**: 技能目录内的辅助文件（脚本、模板、数据文件）的组织与引用方式
- **核心原理**: SKILL.md 中引用相对路径时，Agent 需要将路径解析为技能目录的绝对路径
- **最佳实践**: 脚本放在技能目录的 scripts/ 子目录；模板放在 templates/；SKILL.md 中用 `./scripts/run.sh` 引用
- **真实案例**: openai-image-gen skill 包含 `index.html` 画廊模板 + 生成脚本；Agent 读取 SKILL.md 后知道调用哪个脚本
- **关键参数**: 单技能目录建议 <5MB；脚本需要 shebang 行 + 可执行权限
- **陷阱**: 硬编码绝对路径导致跨环境部署失败；应始终用相对路径

### A04-2804: 技能参数注入与上下文传递
- **定义**: 用户请求中的参数如何传递给技能执行流程
- **核心原理**: Agent 技能无显式参数绑定（不同于函数调用）；参数通过自然语言在上下文中隐式传递，由 Agent 在 SKILL.md 指令中解析
- **模式对比**: 显式参数 (function calling) vs 隐式参数 (skill prompt) — 技能适合复杂多步任务，function calling 适合原子操作
- **真实案例**: weather skill 从用户消息中提取城市名，不需要显式 `city` 参数定义
- **关键参数**: 必须参数应在 SKILL.md 中明确"如果用户未提供 X，则询问"
- **陷阱**: 假设用户总会提供所有信息；防御性编写应包含缺失参数的处理逻辑

### A04-2805: 技能发布与版本管理 (ClawHub)
- **定义**: 将本地开发的技能打包发布到 ClawHub 市场的流程与版本策略
- **核心原理**: `openclaw skills publish` 命令将技能目录上传到 ClawHub；版本号遵循 SemVer
- **发布流程**: 1) 本地测试通过 2) 更新 SKILL.md 版本号 3) `openclaw skills publish <path>` 4) 验证在 ClawHub 可搜索
- **真实案例**: Sandbot 发布了 agent-optimizer、input-validator、github-ops 三个技能到 ClawHub
- **关键参数**: 技能名全局唯一；description 影响搜索排名；README.md 影响安装决策
- **陷阱**: 发布后修改技能名会导致已安装用户断链；应通过版本号迭代而非重命名

### A04-2806: 技能组合与编排模式
- **定义**: 多个技能协作完成复杂任务的编排策略
- **核心原理**: Agent 可在单次对话中顺序激活多个技能；但系统限制"不超过一个技能预加载"
- **编排模式**: 1) 链式 (A→B→C) 2) 分支 (if X then A else B) 3) 并发 (A‖B then merge)
- **真实案例**: 用户要求"搜索 Reddit 上的 AI Agent 痛点并写成邮件" → 先激活 reddit-insights 搜索，再激活 email-marketing 撰写
- **关键参数**: 技能间数据传递通过工作区文件（写入→读取）而非内存
- **陷阱**: 技能冲突（两个技能都想写同一个文件）；应在 SKILL.md 中声明输出路径约定

### A04-2807: 技能测试与质量保障
- **定义**: 验证技能在不同场景下正确执行的测试方法
- **核心原理**: 技能测试 = prompt 测试；需要覆盖正常路径 + 边界情况 + 错误路径
- **测试维度**: 1) 触发准确性 (description 匹配) 2) 执行正确性 (步骤无遗漏) 3) 输出格式 (符合预期) 4) 错误恢复 (依赖不可用时的降级)
- **真实案例**: weather skill 测试用例：正常城市、不存在的城市、中文城市名、只说"天气"不说城市
- **关键参数**: 建议每个技能至少 5 个测试用例；覆盖 happy path + 2 个 edge case + 2 个 error case
- **陷阱**: 只测试 happy path；真实用户的输入千奇百怪

### A04-2808: 技能安全与权限边界
- **定义**: 技能执行时的权限控制与安全约束
- **核心原理**: 技能继承 Agent 的工具权限；SKILL.md 中的指令不能突破 openclaw.json 的 tools.deny 配置
- **安全层级**: 1) 只读技能 (搜索/分析) 2) 写入技能 (文件操作) 3) 外部交互技能 (API 调用/消息发送) 4) 系统操作技能 (exec/配置修改)
- **真实案例**: input-validator skill 检测恶意内容但不执行任何写入操作，属于纯只读技能
- **关键参数**: tools.deny 黑名单优先级 > 技能指令；技能无法绕过工具策略
- **陷阱**: 技能指令中包含 `exec rm -rf` 等危险命令；应在 SKILL.md 中明确标注风险操作并要求用户确认

### A04-2809: 技能国际化与多语言支持
- **定义**: 技能在不同语言环境下的适配策略
- **核心原理**: SKILL.md 本身用一种语言编写；但 description 和输出格式可根据用户语言动态调整
- **策略**: 1) SKILL.md 用英文（最大兼容性） 2) 输出语言跟随用户（"回复使用用户的语言"） 3) 关键术语保留英文原文
- **真实案例**: Sandbot 的技能 SKILL.md 混用中英文；description 用英文确保全球可搜索
- **关键参数**: ClawHub 搜索目前不支持跨语言；英文 description 覆盖面最广
- **陷阱**: 全中文 description 导致英文用户搜不到；全英文 SKILL.md 导致中文 Agent 理解偏差

### A04-2810: 技能性能优化与 Token 经济
- **定义**: 减少技能执行的 token 消耗和延迟的优化方法
- **核心原理**: SKILL.md 的长度直接影响每次激活的 token 成本；精简指令 = 省钱
- **优化策略**: 1) SKILL.md 控制在 500 行以内 2) 大段参考数据放外部文件按需读取 3) 避免重复说明（引用而非复制） 4) 步骤编号精简
- **数据**: 1000 token 的 SKILL.md ≈ ¥0.002/次激活 (qwen3.5-plus)；每日 100 次激活 = ¥0.2/天
- **真实案例**: agent-optimizer skill 的 SKILL.md 从初版 800 行精简到 300 行，token 成本降低 62%
- **关键参数**: 理想 SKILL.md 长度 200-500 行；超过 800 行应拆分
- **陷阱**: 过度精简导致指令模糊，Agent 执行偏差；精简 ≠ 删减关键步骤
