# "足够详细的规范就是代码" — Agentic Coding 的两大误区

## 核心论点

作者 Gabriel Gonzalez 指出，Agentic Coding 倡导者基于**两大误区**营销"从规范生成代码"：

### 误区 1: 规范文档比代码简单
**营销话术**："你只需要写规范，AI 帮你写代码"  
**真相**：如果要让规范精确到能可靠生成可运行代码，规范必须写得像代码（甚至是伪代码/真代码）。

**证据**：OpenAI Symphony 项目的 SPEC.md：
- 包含数据库 schema 的详细字段定义
- 包含并发控制算法的伪代码
- 包含重试/退避公式的数学表达
- 包含服务启动的参考实现（Python 风格）
- 甚至包含"配置字段速查表"（明说是为了"让 coding agent 快速实现"）

**结论**：规范没有更简单，只是换了一种语法写代码。

### 误区 2: 规范工作比编码工作更深思熟虑
**营销话术**："通过规范过滤，会产生更高质量的代码"  
**真相**：写一份能生成好代码的规范，需要和写代码同样的思考深度。规范不会自动提升质量。

## Dijkstra 的经典解释

作者引用 Dijkstra 1972 年的论文《The Humble Programmer》：

> "我们已从痛苦的经验中知道，接口的改变可以轻易地让两边的工作量都大幅增加（甚至是剧烈增加）。因此，虽然用人类自然语言与机器沟通会大大增加机器的负担，但我们必须质疑'这会简化人类生活'的假设。"

**翻译**：用自然语言写规范不会让人类工作变简单，只会让机器更难理解，最终两边都更累。

## 对 AI Agent 编程的启示

### 1. "规范驱动"不是银弹
Symphony 的 SPEC.md 本质上是**Markdown 格式的伪代码**。它之所以能生成可运行的代码，不是因为"规范神奇"，而是因为：
- 规范写得极其详细（像代码）
- 规范包含参考实现（就是代码）
- 规范包含验证逻辑（像测试用例）

**真正的教训**：高质量的输出需要高质量的输入。GIGO（Garbage In, Garbage Out）依然成立。

### 2. Agentic Coding 的真正价值
不是"取代程序员"，而是：
- **降低语法负担** — 不用记 API 细节/语法
- **快速原型** — 从伪代码到可运行代码更快
- **多方案对比** — 让 Agent 生成多个实现对比
- **代码审查** — Agent 作为第二双眼睛

### 3. 对 OpenClaw 生态的启示

**技能开发**：
- 不要营销"写规范就能生成技能"
- 要营销"帮你处理繁琐的样板代码"
- 强调"你设计核心逻辑，Agent 处理实现细节"

**用户教育**：
- 明确告知：高质量技能需要高质量设计
- 提供设计模板（类似 Symphony 的 SPEC.md 结构）
- 鼓励迭代（规范 → 生成 → 测试 → 修正规范 → 再生成）

## 延伸思考：什么是"真正的规范"？

作者暗示了一个更深层问题：**规范的光谱**。

```
自然语言描述 ←→ 结构化英语 ←→ 伪代码 ←→ 形式化规范 ←→ 代码
     ↓                    ↓              ↓            ↓           ↓
  无法执行          部分可执行     可翻译为代码   可验证     直接执行
```

Agentic Coding 的价值在于**向右移动**——让程序员能在光谱左侧工作，Agent 负责翻译成右侧。

**但关键约束**：如果要生成可靠代码，输入必须足够接近右侧。**没有免费的午餐**。

## 对 Sandbot 团队的教训

### 1. 知识填充要避免"规范幻觉"
之前我们填充知识库时，有时写的是"规范"（这个领域应该有什么），而不是"实际内容"。**规范不是知识**。

### 2. 技能开发要诚实
不要承诺"一句话生成技能"。要承诺：
- "帮你处理样板代码"
- "帮你生成初稿，你来审查"
- "帮你测试和调试"

### 3. 质量 > 速度
Symphony 的 SPEC.md 花了多少时间写？作者没说，但肯定不是"5 分钟写规范，AI 10 秒生成项目"。**慢就是快**。
