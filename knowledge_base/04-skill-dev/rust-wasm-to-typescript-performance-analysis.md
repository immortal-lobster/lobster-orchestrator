## 📋 核心事实

### 背景
OpenUI 团队将 openui-lang 解析器从 Rust+WASM 重写为 TypeScript，结果:
- **性能提升**: 2.2-4.6 倍 (单次调用)
- **流式解析提升**: 2.6-3.3 倍 (总成本)
- **代码复杂度**: 降低 (无需 WASM 绑定)

### 技术栈对比
| 方案 | 单次解析 (µs) | 流式总成本 (µs) | 相对性能 |
|------|---------------|-----------------|----------|
| WASM + JSON 往返 | 20.5-61.4 | 基准 | 1.0x |
| WASM + serde-wasm-bindgen | 22.5-79.4 | - | 0.71-0.91x (更慢!) |
| TypeScript (朴素重解析) | 9.3-19.4 | 69-840 | 2.2-4.6x |
| TypeScript (增量缓存) | 9.3-19.4 | 69-255 | **3.0-4.6x** |

## 🔍 深度技术分析

### 1. WASM 边界开销真相
**直觉**: Rust 快，WASM 接近原生速度，应该比 JS 快。

**现实**: WASM 边界开销 dominates 实际计算时间。

**WASM 调用开销分解**:
```
JS 调用 wasmParse(input)
  ↓
[开销 1] 复制字符串：JS heap → WASM linear memory (memcpy)
  ↓
Rust 解析 (确实快，但这部分占比小)
  ↓
[开销 2] 序列化结果：serde_json::to_string()
  ↓
[开销 3] 复制 JSON 字符串：WASM → JS heap (memcpy)
  ↓
[开销 4] JS 反序列化：JSON.parse(jsonString)
  ↓
返回 ParseResult
```

**关键洞察**: 
- Rust 解析本身可能只需 5µs
- 但边界开销固定 15-50µs
- 总时间 = 5µs + 50µs = 55µs (边界占 90%!)

### 2. serde-wasm-bindgen 为什么更慢
**直觉**: 直接返回 JS 对象，跳过 JSON 序列化，应该更快。

**现实**: 30% 更慢。

**原因**:
```
serde-wasm-bindgen 需要:
  - 递归遍历 Rust 结构体每个字段
  - 每个字段都要跨越 WASM-JS 边界转换
  - 一个包含 10 个字段的对象 = 10 次边界 crossing

JSON 方案:
  - serde_json::to_string() 在纯 Rust 中完成 (零边界)
  - 一次 memcpy 复制整个 JSON 字符串
  - V8 的 C++ JSON.parse 一次性处理
  - 总边界 crossing 次数：1 次
```

**教训**: "少次大操作"胜过"多次小操作"。

### 3. 算法优化 > 语言优化
**最大性能提升来源**: 不是 Rust→TS，而是 O(N²)→O(N)。

**朴素流式解析问题**:
```
Chunk 1: parse("root = Root([t") → 14 chars
Chunk 2: parse("root = Root([tbl])\\ntbl = T") → 27 chars
Chunk 3: parse(累积字符串) → ...

1000 字符输出，20 字符 chunk:
  - 50 次 parse 调用
  - 累积处理 25,000 字符
  - O(N²) 复杂度
```

**增量缓存优化**:
```
State: { buf, completedEnd, completedSyms, firstId }

On each push(chunk):
  1. 扫描 buf 从 completedEnd 找 depth-0 换行符
  2. 每个完整语句：解析 + 缓存 AST → 前进 completedEnd
  3. 末尾未完成语句：autoclose + 重新解析
  4. 合并缓存 + 待处理 → resolve + map → 返回

完成的语句永不重新解析，只有末尾进行中语句每 chunk 重解析。
O(total_length) 替代 O(N²)。
```

**性能对比**:
| Fixture | 朴素 TS | 增量 TS | 提升 |
|---------|---------|---------|------|
| contact-form | 316µs | 122µs | 2.6x |
| dashboard | 840µs | 255µs | 3.3x |

## 🎯 对 OpenClaw 的启示

### 1. 性能优化优先级
OpenUI 的经验给出了性能优化的正确顺序:

```
第一优先：算法复杂度优化 (O(N²)→O(N))
  - 影响：3x 提升
  - 成本：中等 (需要重构)

第二优先：减少边界 crossing
  - 影响：2-4x 提升
  - 成本：低 (架构调整)

第三优先：语言/运行时选择
  - 影响：1.5-2x 提升
  - 成本：高 (重写代码)

第四优先：微优化
  - 影响：<1.5x
  - 成本：低 (局部调整)
```

**OpenClaw 应用**:
```
✅ 当前瓶颈：模型调用延迟 (秒级)
✅ 优化方向：
  - 减少调用次数 (批量处理)
  - 增加单次调用内容 (充分利用 1M 上下文)
  - 本地缓存 (搜索结果、网页内容)
❌ 不要过早优化：
  - 工具调用开销 (毫秒级，相对模型调用可忽略)
  - 文件读写开销 (毫秒级)
```

### 2. WASM 的适用场景
OpenUI 总结的 WASM 适用/不适用场景:

**✅ 适合 WASM**:
- 计算密集型 + 最小互操作 (图像处理、加密、物理模拟)
- 大输入 → 标量输出
- 原地修改 (in-place mutation)
- 边界 crossing 频率低
- 移植 C/C++ 库 (SQLite、OpenCV、libpng)

**❌ 不适合 WASM**:
- 解析结构化文本为 JS 对象 (序列化成本不可避免)
- 高频调用小输入函数 (边界开销无法摊销)
- 解析计算本身很快 (V8 JIT 消除语言优势)

**OpenClaw 应用**:
```
⚠️ 考虑过用 WASM 优化知识检索？
  - 知识检索 = 文本解析 + 匹配
  - 属于"❌ 不适合"类别
  - 边界开销会 dominates

✅ 正确优化方向:
  - 索引优化 (倒排索引、向量索引)
  - 缓存策略 (热点知识缓存)
  - 批量检索 (一次查询多条)
```

### 3. 流式处理的通用模式
OpenUI 的"语句级增量缓存"是流式处理的通用模式:

**模式**:
```
1. 识别"不可变边界" (depth-0 换行符)
2. 缓存已完成部分
3. 只重新解析未完成部分
4. 合并结果
```

**OpenClaw 应用**:
```
场景：Cron 多轮知识获取
  - 每轮独立，无需增量
  - 但可缓存已抓取网页

场景：子 Agent 协作
  - 主 Agent 缓存子 Agent 结果
  - 避免重复调用

场景：知识检索
  - 缓存热门搜索查询结果
  - 增量更新索引
```

## 💡 关键教训

### 1. Profile 再优化
OpenUI 团队最初"优化了错误的东西"——以为 Rust 解析慢，实际是边界开销。

**OpenClaw 启示**:
```
✅ 先用 session_status 查看实际耗时
✅ 识别真正瓶颈 (模型调用 vs 工具调用)
✅ 不要凭直觉优化
```

### 2. 边界开销是隐形杀手
WASM 的边界开销、API 调用的网络延迟、序列化的 CPU 开销——这些都是"隐形"的。

**OpenClaw 启示**:
```
✅ 减少工具调用次数 (批量)
✅ 减少网络请求 (缓存)
✅ 减少序列化 (直接传递对象)
```

### 3. 算法 > 语言
O(N²)→O(N) 带来 3x 提升，Rust→TS 只有 2x。

**OpenClaw 启示**:
```
✅ 优先优化任务编排算法
✅ 优先优化缓存策略
✅ 最后才考虑换模型/换语言
```
