## 属性基测试 vs 传统 E2E 测试

### 传统 E2E 测试
```
✅ 写具体用例："点击按钮 A → 应看到页面 B"
❌ 只测试预想到的路径
❌ 维护成本随功能增长线性上升
❌ 无法发现未预见的边界情况
```

### 属性基测试 (Bombadil)
```
✅ 定义属性/不变量："任何操作序列后，购物车总价 ≥ 0"
✅ 自动生成随机操作序列
✅ 自动发现违反属性的最小反例
✅ 覆盖人类无法预想的边界情况
```

## 技术架构

```
Bombadil 工作流:
1. 定义 UI 属性/不变量
2. 自动探索 UI 状态空间 (随机点击/输入/导航)
3. 检测属性违反
4. Shrinking: 最小化反例到最短操作序列
5. 报告可复现的 bug
```
