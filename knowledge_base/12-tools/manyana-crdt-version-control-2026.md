## 传统 VCS vs CRDT VCS 对比

### 冲突展示革命

**传统 Git 冲突** (两人分支自含函数的文件，一人删函数，另一人加一行)：
```
<<<<<<< left
=======
def calculate(x):
    a = x * 2
    logger.debug(f"a={a}")
    b = a + 1
    return b
>>>>>>> right
```
→ 两个不透明块，需要脑内重建发生了什么

**Manyana 冲突**：
```
<<<<<<< begin deleted left
def calculate(x):
    a = x * 2
======= begin added right
    logger.debug(f"a={a}")
======= begin deleted left
    b = a + 1
    return b
>>>>>>> end conflict
```
→ 每个段落标注了"谁做了什么"。左边删除了函数，右边在中间加了一行。**结构可见，不再猜谜。**

## 技术架构

### 1. Weave 数据结构

```
核心概念：文件状态是一个 Weave（编织）
- 包含文件中曾经存在的每一行
- 每行带元数据：何时添加、何时删除
- 合并不需要找共同祖先或遍历 DAG
- 两个状态输入 → 一个状态输出，始终正确
```

**革命性**：历史存在于结构中，而非从 DAG 重建。

### 2. CRDT 保证

```
最终一致性：
- 合并永远不失败
- 结果与合并顺序无关
- 多分支由多人独立合并，结果相同

行序永久化：
- 两分支在同一位置插入代码，CRDT 选定顺序后永久保持
- 防止冲突段在不同分支上以不同顺序解决
```

### 3. 非破坏性 Rebase

```
传统 rebase：创建虚构历史（假装你的提交在最新 main 之上）
CRDT rebase：获得相同效果，但保留完整历史

实现：在 DAG 中添加"主祖先"注解
好处：激进 rebase 产生无单一共同祖先的拓扑
      → 传统三路合并崩溃
      → CRDT 无所谓——历史在 weave 中，不从 DAG 重建
```
