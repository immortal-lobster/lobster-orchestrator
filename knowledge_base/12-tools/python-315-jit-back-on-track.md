## 核心进展

CPython 3.15 的 JIT 编译器提前一年达到 macOS AArch64 性能目标，提前数月达到 x86_64 Linux 目标：

| 平台 | 相比尾调用解释器 | 相比标准解释器 |
|------|----------------|---------------|
| macOS AArch64 | **+11-12%** | - |
| x86_64 Linux | - | **+5-6%** |
| 范围 | -20% ~ +100%+ | 取决于基准测试 |

### 背景回顾
- **3.13/3.14**: JIT 几乎无加速，有时比解释器更慢
- **2025**: Faster CPython 团队失去主要赞助商
- **2026-03**: JIT 终于回到正轨，提前达成目标

## 成功因素分析

### 1. 社区驱动 (Community Stewardship)
```
核心转变：从企业赞助 → 社区志愿者驱动
关键人物：Ken Jin (志愿者)、Savannah Ostrowski、Mark Shannon、
         Diego Russo、Brandt Bucher
教训：任何一个核心贡献者缺席都可能导致失败
```

### 2. 降低贡献门槛
- **Mega-issues**: 将复杂 JIT 优化分解为简单任务
- **单指令优化**: "尝试优化 JIT 中的单条指令"
- **详细指南**: 无 JIT 经验的 C 程序员也能贡献
- **结果**: 11 位贡献者参与，几乎转换整个解释器

### 3. 关键技术赌注

#### Trace Recording (追踪记录)
- Brandt 说服 Ken Jin 重写 JIT 前端为追踪式
- Ken Jin 出于 "spite-driven development" (怨气驱动开发) 尝试
- 原型 3 天完成，完善花了 1 个月
- 结果：成为最大性能提升来源

#### Tail-Calling Interpreter (尾调用解释器)
- 3.14 中引入的优化
- 为 JIT 提供了更好的基线
- JIT 在此基础上再加速 11-12%

## 技术架构

### JIT 三阶段
1. **前端 (Region Selector)**: 选择热点代码区域
   - 3.15 新增：追踪记录 (trace recording)
2. **中端 (Optimizer)**: 优化中间表示
   - 4 位活跃贡献者 (从 2 位增长)
3. **后端 (Code Generator)**: 生成机器码
   - 基于 copy-and-patch 技术

### Bus Factor 改善
| 阶段 | 之前 | 现在 |
|------|------|------|
| 前端 | 1 人 | 2+ 人 |
| 中端 | 2 人 | 4 人 |
| 后端 | 1 人 | 2+ 人 |

## 对 Python 生态的影响

### 性能路线图
- **3.15**: 5-12% 加速 (已达成)
- **3.16**: 目标 10% 加速 + 自由线程支持
- **长期**: 持续优化，缩小与 PyPy 差距

### 对 AI/ML 的影响
- Python 是 AI/ML 主要语言
- JIT 加速有利于纯 Python 代码 (数据预处理、胶水代码)
- NumPy/PyTorch 等 C 扩展不直接受益
- 但循环密集型纯 Python 代码可获 100%+ 加速

### 对 Sandbot 的意义
- Sandbot 使用 Node.js (已有 V8 JIT)，不直接受益
- 但 Python 脚本工具 (priority_scorer.py 等) 将受益
- 知识点：JIT 开发的社区模式值得 Sandbot 联邦借鉴

## 社区管理经验 (对开源项目的启发)

### 成功模式
1. **拆分复杂问题**: mega-issues → 单指令优化任务
2. **详细贡献指南**: 无经验者也能参与
3. **庆祝小成就**: 每个 PR 都有明确结果
4. **方向感**: 清晰的性能目标 (5% by 3.15, 10% by 3.16)

### 失败模式 (需避免)
1. **单人依赖**: bus factor = 1 是项目杀手
2. **过高门槛**: JIT 历来不适合新贡献者
3. **资金依赖**: 赞助商撤出差点终结项目
