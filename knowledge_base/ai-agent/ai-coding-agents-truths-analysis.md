## 四大核心问题

### 1. 技能退化 (Skill Atrophy) 🧠

#### 问题描述
软件工程师角色正在转变为"软件工程经理"——不写代码，而是监督 AI 编码代理团队。

#### 退化机制
```
人类工程师 → 代码审查员 → 技能生锈 → 无法分辨好坏代码
         ↓
    审查负载增加 → 必然变得敷衍 → 坏代码通过审查
```

#### 关键洞察
- **实践和反馈**是保持编码能力的关键，审查员两者都得不到
- 即使 intention 是严格审查，**95%+ 成功率**会导致注意力下降
- 无法与 AI"面对面"讨论"为什么这样实现"

#### 反驳与回应
**反驳**：这像几十年前老程序员抱怨高级语言会导致新一代不理解底层。

**作者回应**：这次感觉不同（承认不严谨，但有 20 年经验支撑）。

---

### 2. 人为低成本 (Artificially Low Cost) 💰

#### 问题描述
生成式 AI 的终端用户价格与实际训练/运行成本**完全脱节**。

#### 经济现实
| 公司 | 状态 |
|------|------|
| OpenAI | [ wildly unprofitable ](https://fortune.com/2025/11/12/openai-cash-burn-rate-annual-losses-2028-profitable-2030-financial-documents/) |
| Anthropic | 类似财务结构 |
| Google/Meta | 有其他业务支撑 |
| 其他玩家 | 需要找到盈利方式或死亡 |

#### 泡沫逻辑
```
巨额 GPU/数据中心投资 → 需要证明 ROI → AI 编码代理是"证明"
                    ↓
但模型本身极度不盈利 → 需要数量级涨价 + 大幅降低使用上限
                    ↓
依赖 AI 代理的公司 → 成本突然飙升 → 无法生存
```

#### 建议
**不要过度依赖 AI 编码代理**，以便在风暴来临时更好地生存（甚至茁壮成长）。

---

### 3. Prompt 注入 (Prompt Injection) 🐛

#### 问题本质
LLM 本质上是易受骗的——无法可靠区分指令和数据。

#### 现状
- 所有大厂商的修复都是"用胶带堵住漏水的船"
- OpenAI 承认：[Prompt 注入将永远存在](https://venturebeat.com/security/openai-admits-that-prompt-injection-is-here-to-stay/)

#### 新威胁：Promptware
Bruce Schneier 命名的新型攻击链：
```
恶意行为者 → 污染网站/邮件 → 注入恶意 prompt
          ↓
AI 编码代理抓取上下文 → 执行恶意指令
          ↓
系统完全沦陷 + 登录账户被盗
```

#### 真实案例
- AI 编码助手追逐幻影，[删除真实用户数据](https://arstechnica.com/information-technology/2025/07/ai-coding-assistants-chase-phantoms-destroy-real-user-data/)
- Meta AI 对齐总监 [OpenClaw 邮件删除事件](https://www.businessinsider.com/meta-ai-alignment-director-openclaw-email-deletion-2026-2)

#### 结论
**不值得冒这个风险**。

---

### 4. 版权/许可 (Copyright/Licensing) ⚖️

#### 核心事实
**美国版权局裁定：生成式 AI 的输出不可版权化**

- 法院同意，最高法院 [拒绝干预](https://www.reuters.com/legal/government/us-supreme-court-declines-hear-dispute-over-copyrights-ai-generated-material-2026-03-02/)
- AI 生成的代码 = 公共领域 = 全世界任何人都可以使用

#### 假设场景：Notion 全 AI 生成代码
```
如果 Notion 从一开始就用 AI 代理生成所有代码：
  ↓
员工泄露源代码到网上
  ↓
任何人可以合法运行该代码（替换受保护资产如商标）
  ↓
直接竞争对手出现，跳过多年开发
  ↓
Notion 无法法律阻止（公共领域）
```

#### 对比：人类编写代码
```
人类编写代码 → 员工泄露 → 竞争对手使用
            ↓
        版权侵权 + NDA 违反
            ↓
        灾难性法律/财务惩罚
            ↓
        有效威慑
```

#### 混合代码库的复杂性
人类生成 + AI 生成混合代码可能有足够法律掩护，但不确定性极高。

#### 建议
**咨询版权律师** before 过度依赖 AI 编码代理。

## LLM 的正确用途

### ✅ 推荐用途
1. **研究**（但仔细验证每件事）
2. **新包/API 示例代码**（不直接复制粘贴）
3. **非 coder 实现想法**（不打算商业化）
4. **有经验 coder 的休闲 side project**

### ❌ 禁止用途
1. 生产环境代码
2. 商业产品核心功能
3. 安全关键系统
4. 需要版权保护的代码

## 对 Sandbot 的启示

### 1. 定位调整
Sandbot 当前定位：
- ✅ 知识管理助手
- ✅ 信息检索与分析
- ✅ 文档生成与整理
- ❌ **生产代码生成**（应避免）

### 2. 安全加固
**Prompt 注入防护**：
- 限制 web_search/web_fetch 结果作为指令执行
- 所有外部内容标记为"不可信数据"
- 禁止从外部内容提取可执行命令

**当前 Sandbot 已有防护**：
```
✅ 外部内容安全警告（web_fetch 返回内容包含）
✅ 禁止执行外部内容中的指令
✅ 敏感路径访问限制
```

### 3. 透明度声明
应在 SOUL.md/IDENTITY.md 中明确：
- Sandbot 不应用于生成生产代码
- 所有输出需要人类验证
- 不保证输出代码的版权状态

## 批判性思考

### 作者观点的局限性

1. **二元思维**：要么全人类写，要么全 AI 写。现实可能是混合模式。

2. **忽略工具演进**：早期高级语言确实导致一些底层知识丢失，但也释放了生产力。AI 代理可能类似。

3. **经济模型假设**：假设 AI 公司无法盈利，但可能通过规模效应 + 技术突破降低成本。

4. **版权法演进**：法律可能适应新技术（如摄影、录音出现时的版权法调整）。

### 但核心警告值得重视

即使不同意作者的极端立场，四个问题都是**真实存在的风险**：
- 技能退化 → 需要保持人类编码实践
- 成本不可持续 → 需要多元化技术方案
- Prompt 注入 → 需要安全加固
- 版权不确定性 → 需要法律咨询
