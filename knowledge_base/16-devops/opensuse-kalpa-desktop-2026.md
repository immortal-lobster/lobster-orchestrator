## 🎯 核心特性

### 不可变架构
```
根文件系统：只读 Btrfs 快照
应用安装：Flatpak + Distrobox
系统更新：原子更新 (A/B 分区)
回滚能力：启动菜单选择历史快照
```

### 技术栈
| 组件 | 技术 | 说明 |
|------|------|------|
| 基础 | openSUSE MicroOS | 不可变服务器版衍生 |
| 桌面 | KDE Plasma 6 | 最新 Qt6 版本 |
| 包管理 | Zypper + Flatpak | 系统/应用分离 |
| 容器 | Distrobox | 兼容传统 deb/rpm |
| 更新 | Snapper + Btrfs | 快照回滚 |

## 💡 设计理念

### 问题陈述
```
传统 Linux 桌面痛点:
1. 系统更新可能破坏现有配置
2. 应用依赖冲突频繁
3. 回滚困难 (需要备份/重装)
4. 企业部署一致性差
```

### Kalpa 解决方案
```
1. 系统/应用分离
   - 系统：只读，原子更新
   - 应用：Flatpak 沙盒，互不干扰

2. 快照即备份
   - 每次更新自动创建 Btrfs 快照
   - 启动时可选择任意历史版本

3. 容器化兼容
   - Distrobox 运行传统应用
   - 不污染主机系统

4. 企业级部署
   - 镜像一致性 100%
   - 批量更新/回滚
```

## 🛠️ 使用场景

### 场景 1: 企业办公桌面
```
部署:
  1. 创建标准镜像 (预装办公应用)
  2. 批量部署到 1000+ 终端
  3. 集中管理更新策略

运维:
  - 每周自动更新 (周末)
  - 故障自动回滚到上周快照
  - 应用通过 Flatpak 集中分发

收益:
  - IT 运维成本下降 60%
  - 系统故障率下降 80%
  - 用户停机时间减少 90%
```

### 场景 2: 开发者工作站
```
配置:
  - 主机系统：只读，仅基础工具
  - Distrobox: Python/Node/Go 开发环境
  - Flatpak: IDE/浏览器/通讯工具

工作流:
  1. 创建项目专用 Distrobox
  2. 安装项目依赖 (不污染主机)
  3. 项目结束删除容器

收益:
  - 项目隔离 100%
  - 系统稳定性提升
  - 切换技术栈零成本
```

```
部署:

  - 学生账户无 root 权限

管理:
  - 教师可创建临时快照 (保存作业)
  - 学期结束批量重置
  - 软件更新集中推送

收益:
  - 维护工作量减少 95%
```
