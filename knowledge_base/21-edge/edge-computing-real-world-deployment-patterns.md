## 为什么需要边缘计算

核心矛盾：**数据在端侧产生，但算力在云端**。往返延迟 50-200ms 对实时场景不可接受。

### 三个硬指标判断是否需要边缘

| 指标 | 阈值 | 典型场景 |
|------|------|---------|
| **延迟** | <20ms | 自动驾驶、工业控制、AR/VR |
| **带宽** | >1Gbps 原始数据 | 视频监控、IoT 传感器集群 |
| **隐私** | 数据不能出本地 | 医疗影像、金融交易、GDPR 合规 |

不满足任何一条 → 直接用云，别折腾边缘。

## 五种主流部署模式

### 模式 1: CDN 扩展型（最简单）

**原理**: 在 CDN 节点跑轻量计算（Cloudflare Workers / Deno Deploy / Fastly Compute）

**适用**: 
- API 路由 / A/B 测试 / 地理围栏
- 请求级别的简单逻辑（<50ms CPU 时间）
- 全球低延迟响应

**限制**: 
- 无持久化状态（或只有 KV 存储）
- 运行时受限（V8 Isolates，无文件系统）
- 单次请求计算上限（通常 10-50ms CPU）

**真实案例**: 
- Shopify Oxygen：店铺前端 SSR 在 Cloudflare Workers 运行
- Vercel Edge Functions：Next.js 中间件在边缘执行

**成本模型**: 按请求数计费，$0.15-0.50/百万请求

---

### 模式 2: 智能网关型（IoT 核心）

**原理**: 在本地网关设备（树莓派/工控机/NVIDIA Jetson）上跑推理和预处理

**适用**:
- 工厂产线缺陷检测
- 农业传感器数据聚合
- 零售门店客流分析

**典型架构**:
```
传感器 ──→ 边缘网关 ──→ 云端
  │         │ (推理)       │ (训练)
  │         │ (过滤)       │ (存储)
  │         │ (告警)       │ (分析)
  │         ↓              ↓
  └── 本地动作         仪表盘/报告
```

**关键决策**: 
- **模型更新策略**: OTA 推送 vs 手动部署？OTA 必须支持回滚
- **离线容错**: 网络断了多久能自主运行？通常要求 4-72 小时
- **算力选型**: Jetson Orin NX ($400-600) 能跑 YOLOv8 实时推理

**真实案例**:
- AWS IoT Greengrass：Lambda 函数跑在本地设备上
- Azure IoT Edge：Docker 容器跑在网关上
- 海康威视 AI 摄像头：端侧推理 + 云端管理

---

### 模式 3: MEC 电信型（5G 核心）

**原理**: 在电信运营商基站/核心网旁放计算节点（Multi-access Edge Computing）

**适用**:
- 云游戏（延迟 < 10ms）
- V2X 车联网
- 工业 AR 远程指导

**生态现状 (2026)**:
- AWS Wavelength：AT&T/KDDI/Vodafone 基站旁的 EC2 实例
- Azure Edge Zones：与运营商合建的边缘数据中心
- Google Distributed Cloud Edge：零售/制造场景

**真实挑战**:
1. **碎片化**: 每个运营商一套 API，跨运营商需要适配
2. **覆盖有限**: 仅主要城市，农村地区无覆盖
3. **成本高**: 比同配置云 VM 贵 1.5-3 倍
4. **调试难**: 出问题需要运营商配合排查

---

### 模式 4: 分布式集群型（Kubernetes 衍生）

**原理**: 用 K8s 变种管理分散的边缘节点集群

**主流工具**:
| 工具 | 定位 | 资源需求 | 适用规模 |
|------|------|---------|---------|
| K3s | 轻量 K8s | 512MB RAM | 10-1000 节点 |
| KubeEdge | 云边协同 | 256MB (边端) | 大规模 IoT |
| OpenYurt | 阿里开源 | 可选 | 多区域边缘 |
| Akri | 设备发现 | 插件级 | 异构设备 |

**核心难题**:
- **网络不可靠**: 边缘节点随时可能断网，Pod 调度必须考虑亲和性
- **异构硬件**: ARM/x86/GPU 混合，镜像管理复杂
- **安全隔离**: 物理可触达的节点需要硬件级安全（TPM/Secure Boot）
- **升级回滚**: 不能像云端一样随便滚动更新，必须支持离线回滚

---

### 模式 5: 浏览器/端侧型（最新趋势）

**原理**: 在用户设备（浏览器/手机/PC）上跑模型推理

**关键技术**:
- WebGPU / WebNN：浏览器 GPU 加速推理
- ONNX Runtime Web：模型跨端部署
- TensorFlow.js / MediaPipe：端侧 ML Pipeline
- WASM + SIMD：CPU 上的高性能推理

**适用**:
- 实时语音转文字（Whisper.cpp in WASM）
- 图像分类/分割（客户端隐私保护）
- LLM 本地推理（llama.cpp / MLX）

**2026 现状**:
- WebGPU 已在 Chrome/Edge/Firefox 稳定版支持
- Whisper Turbo 端侧推理速度接近实时
- 量化 LLM (4-bit) 在 M 系列 Mac 上可用

## 边缘计算选型决策树

```
你的场景是什么？
│
├── Web 请求级处理 → 模式 1 (CDN 扩展)
│   └── Cloudflare Workers / Vercel Edge
│
├── 本地设备数据处理 → 模式 2 (智能网关)
│   └── Jetson / 树莓派 + MQTT
│
├── 超低延迟 + 运营商网络 → 模式 3 (MEC)
│   └── AWS Wavelength / Azure Edge Zones
│
├── 大规模分布式管理 → 模式 4 (K8s 衍生)
│   └── K3s / KubeEdge / OpenYurt
│
└── 用户设备上推理 → 模式 5 (端侧)
    └── WebGPU / WASM / llama.cpp
```

## 踩坑清单（来自生产环境的教训）

### 1. 别把云架构平移到边缘
- ❌ 微服务 + Service Mesh 在边缘行不通（资源不够）
- ✅ 单进程多功能，减少进程间通信

### 2. 时间同步是大坑
- 边缘节点时钟漂移导致日志排序混乱、证书验证失败
- ✅ 必须配置 NTP/PTP，监控时钟偏差

### 3. 远程调试比开发难 10 倍
- 真实情况：设备在矿井/海上/仓库，无法 SSH
- ✅ 预装远程调试通道（SSH 隧道 / WireGuard）
- ✅ 结构化日志 + 本地缓冲 + 断网重传

### 4. OTA 更新必须原子化
- 更新到一半断电 = 变砖
- ✅ A/B 分区 + 回滚机制（参考 Android OTA）
- ✅ 更新前自动快照

### 5. 安全不是可选项
- 物理可触达 = 攻击面无限大
- ✅ Secure Boot + 磁盘加密 + 远程擦除能力
- ✅ 最小权限原则，不存明文密钥
