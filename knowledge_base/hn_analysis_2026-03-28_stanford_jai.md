## 🎯 核心问题

人们已经在报告：给 AI 工具普通机器访问权限后，**丢失文件、工作树被清空、家目录被擦除**。

在"给 Agent 真实账户权限"和"停下来构建容器/VM"之间存在巨大鸿沟。JAI 填补了这个缺口。

---

## 💡 JAI 是什么

**JAI** = Just Another Isolation (斯坦福安全计算机系统研究组开发)

**一句话**: 一行命令，无镜像、无 Dockerfile，为 AI Agent 提供轻量级沙箱边界。

### 核心特性
```bash
# 使用前
codex "帮我清理一下项目"  # ⚠️ 可能删除整个家目录

# 使用后
jai codex "帮我清理一下项目"  # ✅ 只在沙箱内操作
```

### 三层隔离模式

| 模式 | 家目录 | 进程运行 | 保密性 | 完整性 |
|------|--------|----------|--------|--------|
| **Casual** | Copy-on-write 覆盖层 | 你的用户 | 弱 (大部分文件可读) | 覆盖层保护原件 |
| **Strict** | 空的私有家目录 | 无特权 JAI 用户 | 强 (独立 UID) | 完全隔离 |
| **Bare** | 空的私有家目录 | 你的用户 | 中 (你的 UID，但家目录隐藏) | 完全隔离 |

---

## 🔧 工作原理

```
1. 前缀命令 → jai codex / jai claude / jai bash
2. CWD 保持可写 → 工作目录完全访问
3. 家目录变覆盖层 → 修改被 copy-on-write 捕获，原件不动
4. 其他全锁定 → /tmp 私有，其他文件只读
```

### 与传统方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Docker** | 可复现、镜像化环境 | 重、不适合临时沙箱 |
| **bubblewrap** | 强大的命名空间沙箱 | 需要显式组装文件系统视图，容易变成长脚本 |
| **chroot** | 简单 | **不是安全机制**，无挂载隔离 |
| **JAI** | 一行命令、零配置、轻量 | 不是完美安全，不适合多租户/强敌场景 |

## 📊 对 Sandbot 的启示

### 1. 安全边界设计
```
当前 OpenClaw 配置:
- 工作区：/home/node/.openclaw/workspace/
- 密钥：/home/node/.openclaw/secrets/
- 系统路径：~/.ssh, ~/.aws (应禁止访问)

JAI 思路:
- 工作区完全访问 ✅
- 密钥目录只读或隐藏 ⚠️ 需检查
- 系统路径锁定 ✅
```

### 2. 子 Agent 沙箱化
```
7 子 Agent 联邦架构可以借鉴 JAI:
- TechBot → 只允许写 skills/ 和 knowledge_base/
- FinanceBot → 只允许读 data/，禁止写系统文件
- DevOpsBot → 需要 exec 权限，但应限制路径范围
```

### 3. 最小权限原则
```
当前: exec 工具可以运行任何命令
建议: 
- 允许：npm run *, make *, git *, 工作区内操作
- 禁止：rm -rf, curl 到外部，读取 .env/secrets/
- 询问：其他所有命令
```

## 📝 总结

**JAI 的核心价值**: 在"完全信任"和"完全隔离"之间提供实用中间地带。

**对 AI Agent 开发的启示**:
- 安全不应该是"全有或全无"
- 轻量级沙箱比完美容器更容易被采用
- 默认安全比事后审计更有效

**Sandbot 下一步**: 检查当前权限配置，考虑为子 Agent 实现类似 JAI 的沙箱机制。
