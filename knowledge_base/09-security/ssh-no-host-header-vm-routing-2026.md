## 1. 核心问题：SSH 的 IPv4 共享困境

### HTTP 如何解决多站点共享 IP
HTTP/1.1 引入 Host 头部，TLS 引入 SNI 扩展：
```
GET / HTTP/1.1
Host: example.com    ← 这告诉代理转发到哪个后端
```
反向代理读取 Host 头 → 路由到正确的后端服务器。一个 IP 可以托管无限多网站。

### SSH 的困境
SSH 协议在密钥交换阶段**没有任何等价于 Host 头的机制**：
- 客户端连接到 IP:22
- 开始密钥交换
- **代理无法知道客户端想连接哪个 VM**
- 直到认证完成才知道用户身份

### 为什么不能每个 VM 一个 IPv4
- IPv4 地址稀缺且昂贵 ($30-50/年/IP)
- 订阅制 VM 服务（如 exe.dev）提供大量 VM
- 每 VM 一个 IP 会"炸掉订阅成本"

### 为什么不能只用 IPv6
- 部分互联网仍无法通过 IPv6 访问
- 纯 IPv6 = 丢失部分用户的 Web 可达性

## 2. exe.dev 的解法：{User, IP} 元组路由

### 核心思路
不是一个 IP 给所有 VM，而是维护一个 **IPv4 地址池**：

```
规则: 每个用户的每个 VM 获得一个唯一 IP（相对于该用户）
同一 IP 可被不同用户的 VM 共用
{公钥 → 用户} + {用户, IP → VM} = 唯一路由
```

### DNS 结构
```bash
$ dig undefined-behavior.exe.xyz
# ANSWER:
# undefined-behavior.exe.xyz → CNAME → s003.exe.xyz → A → 16.145.102.7
```

- `s003` 是地址池中的一个槽位
- 该 IP 被多个 VM 使用，但每个用户只有一个 VM 用这个 IP

### 认证流程
```
SSH 连接 → 公钥认证 → 公钥识别用户 → {用户, 目标IP} → 唯一 VM
```

### 工程挑战
1. **IP 分配算法**: 创建 VM 时必须根据用户/团队精确分配 IP
2. **代理获取本地 IP**: 裸机简单，云环境中公网 IP 经过 NAT 映射到私有 VPC 地址
3. **跨系统通信**: VM 创建系统 ↔ SSH 代理 ↔ DNS 系统需要实时同步
4. **不可通用**: 需要定制化管理软件，非通用 SSH 多路复用方案

## 3. 替代方案对比

### 3.1 SSH 端口映射
```
VM1 → IP:2201
VM2 → IP:2202
VM3 → IP:2203
```
- ✅ 简单
- ❌ 用户需要记住端口号
- ❌ 不符合 exe.dev 的"统一域名"体验目标

### 3.2 SSH over WebSocket/HTTPS
```
客户端 → HTTPS (带 Host 头) → 代理 → SSH → VM
```
- ✅ 利用 HTTP 的 Host 头路由
- ❌ 需要客户端特殊配置（ProxyCommand）
- ❌ 额外延迟

### 3.3 PROXY Protocol
- 在 SSH 连接前注入 PROXY Protocol 头
- ❌ 标准 SSH 客户端不支持

### 3.4 Tailscale/WireGuard VPN
```
客户端 → VPN → 内网 IP → VM
```
- ✅ 每个 VM 有唯一内网 IP
- ❌ 需要安装 VPN 客户端
- ❌ 增加用户摩擦

## 4. 安全分析

### 4.1 攻击面
- **公钥伪造**: 如果攻击者获得用户公钥，可以路由到该用户的 VM
- **IP 池耗尽**: 大量用户可能导致 IP 池不足
- **DNS 缓存攻击**: CNAME → A 记录链可能被 DNS 缓存投毒

### 4.2 缓解措施
- SSH 公钥认证本身的安全性（私钥不离开客户端）
- IP 池动态扩展机制
- DNSSEC 签名验证

## 5. 对 Agent 基础设施的启示

### 5.1 Agent SSH 远程执行
AI Agent 需要 SSH 到远程 VM 执行代码时面临同样问题：
- 多租户共享 IP 池
- 快速路由到正确的沙箱

### 5.2 Zeroboot + SSH 路由
结合 Zeroboot 的亚毫秒 VM 启动 + exe.dev 的 SSH 路由：
- Agent 发起 SSH → 0.79ms 启动 VM → 路由到该 VM
- 真正的"按需沙箱"

### 5.3 协议级创新需求
SSH 协议可能需要类似 SNI 的扩展来支持多租户场景。
这是一个潜在的标准化机会。

## 6. 关键洞察

### 洞察 1: 协议设计决定基础设施成本
HTTP 的 Host 头是一个看似简单的设计决策，但它使得整个 CDN/反向代理行业成为可能。SSH 缺少这个机制，直接导致了 IPv4 成本问题。

### 洞察 2: 约束驱动创新
exe.dev 不能用"简单但贵"的方案（每 VM 一 IP），被迫发明了 {User, IP} 元组路由。约束是创新之母。

### 洞察 3: 统一体验的工程代价
"域名既可以浏览器打开，也可以 SSH" 这个简单的用户体验目标，背后是复杂的跨系统工程。
