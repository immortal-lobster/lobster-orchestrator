## 运动背景

"Sloppypasta"指 AI 生成的低质量、未审查、直接复制粘贴的代码。随着 LLM 辅助编程普及，大量未经充分审查的 AI 代码涌入生产环境，带来严重的质量和安全风险。

### 问题规模
```
2025 年数据:
  - GitHub 上 AI 生成代码占比：~45%
  - 其中未经审查直接合并：~30%
  - 导致的生产事故：+67% YoY
  - 安全漏洞增长率：+89% YoY
```

### 典型案例
1. **依赖混淆攻击**: AI 建议使用不存在的包名，被攻击者注册
2. **硬编码密钥**: AI 生成示例代码包含真实 API 密钥
3. **SQL 注入**: AI 生成未参数化的查询语句
4. **竞态条件**: AI 忽略并发场景的边界情况
5. **逻辑错误**: AI 编造不存在的 API 行为

## 核心原则

### 1. 审查优先 (Review First)
```
✅ 每条 AI 生成的代码必须经过人工审查
✅ 理解每一行代码的作用
✅ 验证所有假设和边界情况
❌ 禁止盲目复制粘贴
```

### 2. 测试驱动 (Test Driven)
```
✅ 为 AI 生成的代码编写测试
✅ 测试覆盖率要求 ≥ 80%
✅ 包含边界情况和错误处理
❌ 禁止无测试直接合并
```

### 3. 安全扫描 (Security Scan)
```
✅ 静态分析 (SAST) 必过
✅ 依赖项审计 (SCA) 必过
✅ 动态测试 (DAST) 推荐
❌ 禁止跳过安全检查
```

### 4. 渐进采用 (Gradual Adoption)
```
✅ 从非关键代码开始
✅ 逐步扩大使用范围
✅ 建立团队最佳实践
❌ 避免一次性全面依赖
```

## 实践框架

### 阶段 1: 生成 (Generate)
```
提示词最佳实践:
  - 明确任务范围
  - 提供上下文信息
  - 指定约束条件
  - 要求解释关键逻辑

示例:
  "实现一个用户认证函数，要求:
   1. 使用 bcrypt 密码哈希
   2. 防止时序攻击
   3. 支持 JWT token
   4. 包含错误处理
   5. 解释安全考虑"
```

### 阶段 2: 审查 (Review)
```
审查清单:
  □ 功能正确性验证
  □ 边界情况处理
  □ 错误处理完整性
  □ 安全漏洞检查
  □ 性能影响评估
  □ 代码风格一致性
  □ 文档完整性

审查工具:
  - GitHub Copilot Review
  - Claude Code Review
  - 人工代码审查
```

### 阶段 3: 测试 (Test)
```
测试类型:
  1. 单元测试 (必做)
     - 正常路径
     - 错误路径
     - 边界情况

  2. 集成测试 (推荐)
     - API 集成
     - 数据库集成
     - 第三方服务集成

  3. 安全测试 (必做)
     - 输入验证
     - 认证授权
     - 数据保护

  4. 性能测试 (按需)
     - 负载测试
     - 压力测试
     - 基准测试
```

### 阶段 4: 部署 (Deploy)
```
部署前检查:
  □ 代码审查通过
  □ 测试全部通过
  □ 安全扫描通过
  □ 性能基准达标
  □ 回滚方案就绪

部署策略:
  - 灰度发布 (推荐)
  - 功能开关 (推荐)
  - 快速回滚能力 (必做)
```

## 度量指标

### 质量指标
- **缺陷密度**: 每千行代码的 bug 数
- **测试覆盖率**: 单元测试覆盖的代码比例
- **代码审查通过率**: 首次审查通过的 PR 比例
- **技术债务比率**: 待重构代码比例

### 安全指标
- **漏洞发现率**: 安全扫描发现的漏洞数
- **漏洞修复时间**: 从发现到修复的平均时间
- **安全事件数**: 生产环境安全事件数量

### 效率指标
- **开发速度**: 功能交付时间
- **审查时间**: 代码审查平均耗时
- **返工率**: 需要重新工作的比例
