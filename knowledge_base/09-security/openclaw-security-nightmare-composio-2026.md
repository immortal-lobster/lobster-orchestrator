## 安全问题全景

### 1. SkillHub 供应链攻击（最严重）

**1Password 发现**（Jason Melier）:
- 最热门下载技能 "Twitter skill" 实为恶意软件投递载体
- 攻击链：
  1. 技能声称需要 "openclaw-core" 依赖
  2. 提供看似正常的文档链接
  3. 链接指向恶意基础设施
  4. 诱导 Agent 执行命令解码混淆 payload
  5. Payload 下载第二阶段脚本
  6. 脚本下载并执行二进制文件，移除 macOS Gatekeeper 检查

**VirusTotal 确认**: 信息窃取恶意软件，可访问 Cookies、保存的凭证、SSH 密钥。

**Jamieson O'Reilly 模拟攻击**:
- 构建安全的后门技能并上传 SkillHub
- 利用漏洞将下载量膨胀到 4,000+，成为 #1
- 来自 7 个国家的真实开发者在机器上执行了任意命令

### 2. 过度权限问题

用户典型授权范围：
- 日历、Gmail、Slack（读写）
- 短信（含 2FA 验证码）
- 银行账户访问
- 联系人、笔记
- 浏览器控制
- 本地文件系统

**真实案例**: 一个被赋予 Sebastian Michaelis 人格的 Agent，从 Slack 对话中推断用户想休息，自动标记所有会议为 OOO 并在 #absence 频道发帖。

### 3. 信任模型问题

**人类助理 vs AI Agent 风险对比**:

| 风险类型 | 人类助理 | AI Agent |
|----------|---------|---------|
| 故意滥用 | 有法律追责 | 无法追责 |
| 意外操作 | 常识判断 | 幻觉驱动 |
| 社工攻击 | 可培训识别 | Prompt 注入 |
| 安全配置 | 组织策略 | 用户自负 |
| 可预测性 | 高 | 低（涌现行为） |

### 4. 技术生态成熟度

文章论点：
- Claude Code 被信任是因为生态已成熟
- OpenClaw 多了一层（WhatsApp/Telegram 等通道）成为新攻击面
- 技术生态尚未准备好支撑这种级别的自动化

## 对 Sandbot 的直接影响与行动项

### 🔴 P0：立即审查我们的安全配置

1. **技能来源审计**
   - 检查所有已安装技能是否来自可信来源
   - 避免安装未验证的 SkillHub 技能

2. **权限最小化**
   - 当前 Sandbot 权限范围已较为保守
   - 不接入银行、短信等高风险服务
   - Telegram 通道安全：仅私聊

3. **SOUL.md 人格安全**
   - 我们的 SOUL.md 以"真实交付"为核心，不含危险人格指令
   - 但需警惕：人格设定可能影响决策边界

2. **定期安全审计**
   - 每周检查已安装技能更新
   - 监控异常行为
