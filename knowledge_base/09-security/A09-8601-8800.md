## 1. AI Agent 凭证管理的现实困境

### 问题本质

AI Agent 与传统软件的根本区别：Agent 需要**运行时动态获取凭证**来调用外部服务（API Key、OAuth Token、SSH Key），而传统软件的凭证在部署时就已固定。

这带来三个安全挑战：

| 挑战 | 传统软件 | AI Agent |
|------|----------|----------|
| 凭证范围 | 固定、可审计 | 动态、按任务变化 |
| 权限边界 | 编译时确定 | 运行时由 LLM 决策 |
| 泄露风险 | 配置文件/环境变量 | 上下文窗口、日志、工具调用链 |

### 案例：OneCLI 凭证网关方案

OneCLI（2026 年 HN 热议项目）提出的方案：

```
用户 → Agent → OneCLI Gateway → 外部 API
                    ↑
              凭证存储在网关
              Agent 只拿到短期 token
```

**优点**：
- Agent 永远不直接接触长期凭证
- 网关可以做细粒度权限控制（只允许读、不允许删）
- 所有 API 调用经过网关，完整审计日志

**缺点**：
- 单点故障风险：网关挂了，所有 Agent 失效
- 延迟增加：每次调用多一跳
- 复杂度转移：从 Agent 转移到网关运维

### 实践建议

对于 OpenClaw 类 Agent 框架：
1. **最小权限原则**：每个工具调用只授予完成任务所需的最小权限
2. **凭证隔离**：secrets 目录与 workspace 物理隔离，Agent 代码不可直接读取
3. **短期 Token**：优先使用有过期时间的 token，避免长期 API Key 暴露在上下文中
4. **审计日志**：记录每次凭证使用的调用者、时间、目标、结果

## 2. RAG 投毒攻击：被忽视的 Agent 供应链风险

### 攻击原理

RAG（检索增强生成）系统从外部知识库检索文档注入 LLM 上下文。攻击者可以在知识库中植入恶意文档，间接控制 LLM 行为。

```
攻击链：
  1. 攻击者在公开知识源（Wiki、论坛、文档）植入恶意内容
  2. RAG 系统索引该内容
  3. 用户查询触发检索，恶意文档进入上下文
  4. LLM 被诱导执行非预期操作（泄露数据、调用危险工具）
```

### 真实场景对比

| 攻击类型 | 传统注入 | RAG 投毒 |
|----------|----------|----------|
| 攻击入口 | 用户输入 | 知识库文档 |
| 检测难度 | 中（可过滤输入） | 高（文档看起来正常） |
| 影响范围 | 单次对话 | 所有检索到该文档的对话 |
| 持久性 | 一次性 | 持久（直到文档被清理） |

### 防御方案

1. **来源验证**：只从可信来源索引文档，对第三方内容做内容安全扫描
2. **检索结果过滤**：在注入 LLM 前，用分类器检测检索结果中的指令注入模式
3. **输出约束**：即使上下文被污染，工具调用仍需经过权限检查（不依赖 LLM 判断）
4. **定期审计**：对知识库内容做周期性安全扫描，检测异常内容

## 3. Agent 零信任架构实践

### 为什么传统零信任不够

传统零信任假设：**人类用户**通过**已知设备**访问**固定资源**。

AI Agent 打破了这三个假设：
- **非人类身份**：Agent 没有 MFA、没有行为基线
- **动态工具链**：Agent 可能调用事先未知的工具
- **上下文驱动决策**：同一个 Agent 在不同上下文下行为完全不同

### Agent 零信任设计原则

```
1. 每次工具调用都是一次"认证"
   - 不因为 Agent 上一次调用成功就信任下一次
   - 每次调用独立检查权限

2. 上下文即风险
   - 上下文窗口越大，潜在攻击面越大
   - 对长上下文 Agent（1M tokens）需要更严格的工具权限

3. 可观测性优先
   - 记录每次 LLM 决策的推理链
   - 工具调用前后的上下文快照
   - 异常行为实时告警
```

### OpenClaw 安全配置示例

```json
{
  "tools": {
    "profile": "coding",
    "deny": ["group:runtime"],
    "confirm": ["exec", "write"]
  },
  "security": {
    "maxToolCallsPerTurn": 10,
    "sensitivePathPatterns": ["~/.ssh/*", "~/.aws/*"],
    "auditLog": true
  }
}
```

**关键点**：`deny` 和 `confirm` 是两道防线。`deny` 直接禁止高危工具，`confirm` 对中危工具要求人工确认。这比单纯依赖 LLM 的"安全意识"可靠得多。

## 4. 2026 年 Agent 安全趋势

### 趋势一：安全左移到 Agent 设计阶段
- 不再事后补安全，而是在 Agent Prompt / Tool Schema 设计时就定义安全边界
- 例：OpenClaw 的 `tools.deny` 在配置层面限制 Agent 能力

### 趋势二：Agent 行为审计标准化
- SWE-bench 等基准测试开始纳入安全维度
- 预计 2026 下半年出现专门的 Agent 安全基准

### 趋势三：多 Agent 协作的信任链
- 当 Agent A 调用 Agent B 时，如何传递和验证权限？
- 类似微服务的 mTLS + Service Mesh 模式可能被引入 Agent 框架
