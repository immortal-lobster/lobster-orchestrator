## 事件概要

2026 年 3 月 22 日，DeFi 协议 Resolv 因**链下基础设施被入侵**损失约 $23M。攻击者入侵了 AWS KMS 环境，获取了特权签名密钥，铸造了 8000 万枚无抵押 USR 稳定币，导致 USR 价格暴跌 80%。

**关键事实**: 智能合约代码完全按设计运行——问题出在信任架构。

## 攻击全流程

### 步骤 1: 入侵 AWS KMS 环境
- 攻击者渗透了 Resolv 的 AWS 云基础设施
- 获取了存储在 KMS 中的 SERVICE_ROLE 签名密钥
- 拥有了与 Resolv 自身铸币操作完全相同的权限

### 步骤 2: 铸造代币
```
攻击者存入 ~$100K-$200K USDC (小额合法存款)
    ↓
调用 requestSwap (存款请求)
    ↓
使用窃取的密钥调用 completeSwap (签名铸币)
    ↓
铸造 50M USR (第一笔交易)
铸造 30M USR (第二笔交易)
    ↓
总计: ~80M USR (远超 $200K 抵押)
```

### 步骤 3: 变现
```
80M USR → wstUSR (质押版本)
    ↓
wstUSR → 其��稳定币 (逐步兑换)
    ↓
稳定币 → ETH (~$25M)
    ↓
USR 价格暴跌 80%，协议被迫停运
```

## 根本原因分析

### 设计缺陷：链下信任 + 无上限铸币

Resolv 的铸币流程：
1. **requestSwap**: 用户存入 USDC，提交铸币请求
2. **completeSwap**: 链下服务 (SERVICE_ROLE 密钥) 审批并执行铸币

**致命问题**:
- ✅ 合约检查了最低 USR 产出
- ❌ **没有检查最高 USR 产出**
- ❌ **没有链上价格预言机**
- ❌ **没有铸币比率限制**
- ❌ **没有单笔铸币上限**

> "Whatever the key holder signs will get minted." — 这就是全部安全模型。

### 信任边界错位

| 组件 | 信任级别 | 实际安全性 |
|------|----------|------------|
| 智能合约 | 高 (链上验证) | ✅ 按设计运行 |
| AWS KMS | 中 (云安全) | ❌ 被入侵 |
| SERVICE_ROLE 密钥 | **无限** (无上限) | ❌ 成为攻击向量 |
| 链下服务 | 隐式信任 | ❌ 无独立验证 |

## 安全教训

### 1. 链上强制约束 > 链下信任
```
❌ 错误: 信任链下签名 → 无限铸币
✅ 正确: 链上硬编码铸币比率上限
✅ 正确: 链上价格预言机交叉验证
✅ 正确: 时间窗口 + 金额限制 (rate limiting)
```

### 2. 最小权限原则
```
❌ 错误: SERVICE_ROLE 可铸造任意数量
✅ 正确: 单笔限额 + 日限额 + 总限额
✅ 正确: 多签 (multisig) 大额铸币
✅ 正确: 时间锁 (timelock) 异常操作
```

### 3. 密钥管理最佳实践
```
❌ 错误: 单一 KMS 密钥控制全部铸币
✅ 正确: 密钥分层 (操作/审批/紧急)
✅ 正确: 硬件安全模块 (HSM) 而非纯云 KMS
✅ 正确: 密钥轮换 + 使用审计
✅ 正确: 异常检测 (单笔金额/频率告警)
```

### 4. 纵深防御
```
即使密钥被盗，仍应有多层防护:
Layer 1: 链上铸币上限 (合约级)
Layer 2: 时间锁 + 多签 (治理级)
Layer 3: 实时链上监控 (检测级)
Layer 4: 紧急暂停机制 (响应级)
Layer 5: 保险 + 储备金 (恢复级)
```
