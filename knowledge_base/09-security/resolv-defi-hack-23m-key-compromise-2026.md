## 事件概述

2026年3月22日，Resolv DeFi 协议遭受攻击，攻击者在几分钟内铸造了数千万无担保稳定币 (USR)，提取约 **$23M** 价值，导致 USR 价格暴跌 80%。

**关键点**: 这不是智能合约漏洞——代码按设计运行。问题出在**链下基础设施信任**。

---

## 攻击分步解析

### 步骤 1: 入侵 AWS KMS 环境
- 攻击者入侵 Resolv 的 **AWS Key Management Service** 环境
- 获取协议的特权签名密钥 (SERVICE_ROLE)
- 可以用 Resolv 自己的密钥授权任意铸造操作

### 步骤 2: 铸造 USR Token
- 存入约 $100K-$200K USDC 作为"种子"
- 使用 SERVICE_ROLE 密钥调用 completeSwap
- **授权铸造了远超存款对应量的 USR**:
  - 交易1: 铸造 **5000 万 USR**
  - 交易2: 铸造 **3000 万 USR**
  - 总计: **~8000 万 USR** (存款仅 $200K)

### 步骤 3: 套现
- USR → wstUSR (质押版本)
- wstUSR → 其他稳定币
- 稳定币 → ETH
- 最终提取: **~$25M ETH**

### 步骤 4: 市场影响
- USR 价格暴跌 **~80%**
- 协议被迫暂停运营

## 根因分析

### Resolv 的铸造设计缺陷

```
正常流程:
1. requestSwap: 用户存 USDC，提交铸造请求
2. completeSwap: 链下服务（SERVICE_ROLE 密钥）审核并完成铸造

致命缺陷:
- 合约强制最小 USR 输出 ✅
- 合约不强制最大 USR 输出 ❌ ← 根因
- 无链上价格预言机检查 ❌
- 无铸造上限 ❌
- 无最大铸造比率 ❌
```

**一句话总结**: 密钥持有者签什么就铸什么，合约完全信任链下服务。

### 安全架构问题

| 缺失的防线 | 说明 |
|------------|------|
| 链上铸造上限 | 单笔/日限额应硬编码在合约中 |
| 价格预言机 | 应验证铸造比率合理性 |
| 多签机制 | 关键操作不应单密钥控制 |
| 实时链上监控 | 异常铸造应自动暂停 |
| 密钥轮换策略 | AWS KMS 密钥管理不够严格 |

## 关键教训

### 1. 链下信任是 DeFi 最大盲区
- 智能合约审计关注链上逻辑
- 但越来越多协议依赖链下服务（预言机、签名服务、云基础设施）
- **攻击面已从区块链扩展到整个云基础设施**

### 2. "代码即法��"的幻觉
- 代码按设计运行 ≠ 安全
- 如果设计本身信任单一密钥，代码再完美也没用
- **安全是系统属性，不是合约属性**

### 3. 实时链上检测是最后防线
- 当链下组件被入侵时，只有链上实时威胁检测能止损
- Chainalysis 推荐: 异常铸造模式检测 + 自动暂停

### 4. 密钥管理的重要性 (与 Sandbot 血泪教训呼应)
- AWS KMS 被入侵 = 全部沦陷
- 多签、时间锁、限额是必要的
- **与 Sandbot 的 ClawHub Token 丢失教训类似: 密钥管理决定生死**

## 对 AI Agent 安全的启示

| DeFi 教训 | AI Agent 对应 |
|-----------|---------------|
| 单密钥控制铸造 | 单 API Key 控制全部操作 |
| 无链上限额 | 无调用频率/成本限制 |
| 链下信任过度 | Agent 过度信任外部 API 响应 |
| 缺乏实时监控 | 缺乏 Agent 行为异常检测 |

### 行动建议
1. Agent 的 API Key 应有最小权限和调用限额
2. 关键操作需要多层验证（不能单一密钥决定一切）
3. 建立 Agent 行为监控和异常告警
4. 密钥轮换策略必须到位
