## 📚 01-ai-agent 领域知识点 (A01-0001 到 A01-0500)

### A01-0001 到 A01-0100: AI Agent 基础概念

**A01-0001: AI Agent 定义**
- **定义**: 能够感知环境、做出决策、执行动作以实现目标的自主智能系统
- **核心要素**: 感知 (Perception)、推理 (Reasoning)、行动 (Action)、学习 (Learning)
- **与传统程序区别**: 自主性、适应性、目标导向、持续学习
- **应用场景**: 个人助理、自动化工作流、智能客服、数据分析
- **关键指标**: 任务完成率、自主决策比例、学习效率、用户满意度
- **代表系统**: AutoGPT、BabyAGI、LangChain Agents、OpenClaw Sandbot

**A01-0002: Agent 架构模式**
- **定义**: Agent 系统的组织结构和工作流程设计
- **常见模式**: 
  - ReAct (Reasoning + Acting): 推理与行动交替进行
  - Plan-and-Execute: 先规划后执行
  - Reflection: 自我反思改进
  - Multi-Agent: 多 Agent 协作
- **选择原则**: 任务复杂度、实时性要求、容错需求
- **性能影响**: 架构决定 70% 以上的 Agent 表现上限

**A01-0003: 提示工程 (Prompt Engineering)**
- **定义**: 设计和优化输入提示以引导 LLM 产生期望输出的技术
- **核心技巧**:
  - Few-shot: 提供示例引导格式
  - Chain-of-Thought: 引导逐步推理
  - Role-playing: 设定角色增强专业性
  - Constraint: 明确限制减少幻觉
- **效果提升**: 优质提示可提升 30-50% 输出质量
- **成本考量**: 长提示增加 token 消耗，需平衡质量与成本

**A01-0004: 工具调用 (Tool Calling)**
- **定义**: Agent 调用外部 API/函数扩展能力边界的机制
- **实现方式**:
  - Function Calling: LLM 原生支持 (GPT-4、Qwen)
  - JSON Schema: 结构化参数定义
  - Dynamic Loading: 运行时加载工具
- **安全考虑**: 权限控制、参数验证、速率限制、审计日志
- **典型工具**: 搜索、文件操作、数据库查询、消息发送

**A01-0005: 记忆系统 (Memory System)**
- **定义**: Agent 存储和检索历史经验以支持持续学习的能力
- **记忆类型**:
  - 短期记忆: 当前会话上下文 (Context Window)
  - 长期记忆: 持久化存储 (向量数据库、文件)
  - 工作记忆: 任务执行中的临时状态
- **检索策略**: 语义搜索、关键词匹配、时间衰减、重要性评分
- **Sandbot 实践**: MEMORY.md (核心) + memory/*.md (每日) + knowledge_base/ (知识)

**A01-0006: 目标管理 (Goal Management)**
- **定义**: Agent 设定、追踪、调整目标以实现复杂任务的机制
- **目标层级**: 愿景 → 战略目标 → 战术目标 → 具体任务
- **追踪方法**: 任务清单、进度百分比、完成验证、障碍记录
- **动态调整**: 根据反馈、资源、优先级变化调整目标
- **Sandbot 实践**: memory/tasks.md (P0/P1/P2 优先级)

**A01-0007: 自我反思 (Self-Reflection)**
- **定义**: Agent 对自身表现进行评估和改进的能力
- **反思时机**: 任务完成后、遇到错误时、定期回顾
- **反思内容**: 什么做得好、什么问题、如何改进、学到什么
- **改进循环**: 执行 → 反思 → 调整 → 再执行
- **Sandbot 实践**: 每日复盘 Cron、血泪教训写入 MEMORY.md

**A01-0008: 幻觉检测 (Hallucination Detection)**
- **定义**: 识别和纠正 LLM 生成虚假/错误信息的能力
- **检测方法**:
  - 事实核查：对照可信来源验证
  - 一致性检查：多轮对话逻辑自洽
  - 置信度评估：要求模型给出确定性
  - 外部验证：文件存在性检查 (ls/cat)
- **Sandbot 教训**: 2026-02-18 的 18 天幻觉循环，设计完美架构但零代码实现

**A01-0009: 成本优化 (Cost Optimization)**
- **定义**: 在保证质量前提下最小化 API 调用成本
- **优化策略**:
  - 批量处理：合并多个任务为一次调用
  - 本地优先：能本地计算不调用模型
  - 缓存复用：重复查询用缓存
  - 模型选择：简单任务用小模型
- **Sandbot 教训**: 2026-03-31 两天调用~10000 次，成本¥50-100+，V6.4.0 设定每日 200 次上限

**A01-0010: 安全边界 (Safety Boundaries)**
- **定义**: Agent 行为的限制和约束机制
- **禁止操作**: 删除用户数据、发送外部消息、访问敏感路径、保存私钥
- **需确认操作**: 批量删除、安装软件、修改配置、发送消息
- **安全原则**: 最小权限、明确同意、审计日志、可回滚
- **Sandbot 实践**: AGENTS.md 安全红线章节

## 📚 03-federal-system 领域知识点 (A03-0001 到 A03-0500)

### A03-0001 到 A03-0100: 联邦系统基础架构

**A03-0001: 联邦系统定义**
- **定义**: 由多个独立但协作的 Agent 组成的分布式智能系统
- **核心特征**: 去中心化、专业化、协作性、可扩展性
- **与单体系统对比**: 
  - 单体：单一 Agent 处理所有任务，简单但能力受限
  - 联邦：多 Agent 各司其职，复杂但能力强大
- **适用场景**: 复杂项目、多领域任务、高并发需求
- **Sandbot 实践**: 7 子 Agent 联邦 (TechBot/FinanceBot/CreativeBot/AutoBot/ResearchBot/Auditor/DevOpsBot)

**A03-0002: Agent 角色设计**
- **定义**: 为每个子 Agent 定义明确的职责和能力边界
- **设计原则**:
  - 专业化：每个 Agent 专注一个领域
  - 互补性：Agent 之间能力互补不重叠
  - 独立性：Agent 可独立完成任务
  - 协作性：Agent 之间可高效协作
- **Sandbot 7 子 Agent**:
  - TechBot 🛠️: 技术教程开发 (ROI: 3.2)
  - FinanceBot 💰: 金融收益分析 (ROI: 2.1)
  - CreativeBot 🎨: 创意内容生成 (ROI: 2.0)
  - AutoBot 🤖: 数据抓取自动化 (ROI: 2.5)
  - ResearchBot 🔬: 深度研究分析 (ROI: 2.5)
  - Auditor 🔍: 质量保障审计 (ROI: 3.0)
  - DevOpsBot ⚙️: 工程化运维 (ROI: 2.0)

**A03-0003: 任务分配机制**
- **定义**: 主 Agent 将任务分配给合适子 Agent 的策略
- **分配策略**:
  - 基于类型：技术任务→TechBot，金融任务→FinanceBot
  - 基于负载：选择当前负载最低的 Agent
  - 基于专长：选择最匹配技能的 Agent
  - 基于 ROI: 选择预期收益最高的 Agent
- **Sandbot 实践**: 主 Agent 分析任务类型 → 分配对应子 Agent → Auditor 审查 → 最终交付

**A03-0004: Agent 间通信**
- **定义**: 子 Agent 之间交换信息和协调工作的机制
- **通信方式**:
  - sessions_send: 发送消息到另一个会话
  - sessions_history: 读取另一个会话历史
  - 共享文件系统：通过 workspace 文件交换数据
  - 消息队列：复杂场景用队列解耦
- **通信协议**: 结构化 JSON、明确的任务描述、完成状态标记
- **Sandbot 实践**: 主 Agent 通过 sessions_spawn 调用子 Agent，通过 subagents list 监控状态

**A03-0005: 质量保障机制**
- **定义**: 确保联邦系统输出质量的流程和标准
- **QA 流程**:
  - 子 Agent 自检：完成任务后自我审查
  - Auditor 审查：独立 Agent 进行质量审计
  - 主 Agent 终审：最终交付前确认
  - 用户反馈：根据反馈持续改进
- **质量标准**: 真实性 (可验证文件)、完整性 (覆盖需求)、准确性 (无错误信息)、时效性 (及时交付)
- **Sandbot 实践**: Auditor 子 Agent 专职质量审计，ROI>3.0

**A03-0006: 负载均衡**
- **定义**: 在多个 Agent 之间合理分配工作负载
- **负载指标**: 任务队列长度、CPU/内存使用率、响应时间、错误率
- **均衡策略**:
  - 轮询：依次分配给每个 Agent
  - 最少连接：分配给当前任务最少的 Agent
  - 加权：根据 Agent 能力分配不同权重
  - 动态：根据实时负载动态调整
- **Sandbot 配置**: 主 Agent maxConcurrent: 4, 子 Agent maxConcurrent: 8

**A03-0007: 故障恢复**
- **定义**: Agent 失败时的自动恢复和容错机制
- **故障类型**: 超时、错误、资源不足、网络问题
- **恢复策略**:
  - 重试：失败后自动重试 (最多 3 次)
  - 降级：用简化版本完成任务
  - 转移：将任务分配给其他 Agent
  - 告警：严重故障通知用户
- **Sandbot 实践**: sessions_spawn 设置 timeoutSeconds，超时自动失败，记录到 memory/

**A03-0008: 状态管理**
- **定义**: 追踪和管理联邦系统整体状态
- **状态类型**:
  - Agent 状态：空闲、忙碌、错误、离线
  - 任务状态：待分配、进行中、已完成、已失败
  - 系统状态：健康、警告、严重
- **状态存储**: 内存 (实时)、文件 (持久化)、数据库 (复杂查询)
- **Sandbot 实践**: subagents/ 目录下每个 Agent 有 SOUL.md 配置文件，sessions_list 查看运行状态

**A03-0009: 性能监控**
- **定义**: 持续追踪联邦系统性能指标
- **关键指标**:
  - 任务完成率：成功完成的任务比例
  - 平均响应时间：从任务分配到完成的时间
  - ROI: 实际收益/模型调用成本
  - 用户满意度：反馈评分
- **监控工具**: 日志分析、指标收集、告警系统、仪表板
- **Sandbot 实践**: 每日复盘 Cron、心跳检查、memory/tasks.md 追踪进度

**A03-0010: 扩展性设计**
- **定义**: 系统支持增加新 Agent 和能力的设计
- **扩展方式**:
  - 水平扩展：增加同类型 Agent 实例
  - 垂直扩展：增加新类型 Agent
  - 能力扩展：为现有 Agent 增加新技能
- **扩展原则**: 松耦合、标准化接口、向后兼容、文档化
- **Sandbot 实践**: 7 子 Agent 配置就绪，可随时启用；skills/ 目录支持新增技能
