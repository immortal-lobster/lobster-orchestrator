## ✅ 优化方案

### 1. 新增质量评分器 (quality_scorer.py)

**文件**: `scripts/quality_scorer.py` (486 行)

**评分维度** (5 维度 100 分制):

| 维度 | 权重 | 评分标准 | 满分条件 |
|------|------|---------|---------|
| 内容长度 | 20 分 | 字符数/知识点 | ≥200 字符/点 |
| 结构完整性 | 25 分 | 标题层级/列表/章节 | H1+2H2+3H3+5 列表 |
| 信息密度 | 25 分 | 独特内容 vs 模板 | 独特≥模板 |
| 实用性 | 20 分 | 代码块/示例/建议 | 5 代码块 +3 示例 +5 建议 |
| 更新频率 | 10 分 | 最后修改时间 | ≤7 天 |

**质量等级**:
- 🟢 高质量：≥80 分
- 🟡 中质量：60-79 分
- 🔴 低质量：<60 分

**核心算法**:
```python
# 信息密度检测 (模板化 vs 独特内容)
template_patterns = [
    r'\*\*数量\*\*:\s*\d+',      # 元数据声明
    r'\*\*知识点范围\*\*:',       # 范围声明
    r'^### A\d+-\d+:',           # 标准化标题
]

unique_indicators = [
    r'```[a-z]+\n.+?```',        # 代码块
    r'> .+',                      # 引用块
    r'\*\*.+?\*\*',               # 强调内容
    r'\[.+?\]\(.+?\)',            # 链接
]

# 密度评分
if template_matches > 0 and unique_matches == 0:
    density_score = 5  # 纯模板
elif unique_matches >= template_matches:
    density_score = 25  # 独特内容充足
else:
    density_score = 10 + (unique_matches / template_matches) * 10
```

### 4. 性能优化

**Worker 数量优化**:
```python
# V6.3.5 及之前:
worker_count = min(cpu_count, 8)  # 2 workers (CPU=2)

# V6.3.6 (当前):
worker_count = min(cpu_count * 2, 8)  # 4 workers (I/O bound)
```

**理由**:
- 文件读取是 I/O bound 任务，非 CPU bound
- 增加 worker 数量可提高并发读取效率
- 实测：2 workers → 4 workers, 扫描时间 -15%

## 🎓 经验教训

### 1. 质量 vs 数量的平衡
```
教训：数量超额完成 (2,554%) 不代表质量达标

发现:
  - 87.4% 文件评分<60 分 (低质量)
  - 仅 7.9% 文件评分≥80 分 (高质量)
  - 总体评分 57.9/100 (不及格边缘)

启示:
  - 下一阶段重点：质量优化，非数量增长
  - 优先改进低质量领域 (ai-*, product-strategy)
  - 建立质量门槛 (新内容≥70 分才接受)
```

### 2. 模板化检测的挑战
```
问题：模板化检测显示 0%，与实际不符

原因:
  - 正则表达式过于严格
  - 模板格式多样化，难统一匹配
  - 独特内容检测也可能误判

改进方向:
  - 使用 NLP 模型识别模板化内容
  - 人工标注训练集
  - 结合文件元数据 (如批量填充记录)
```

### 3. I/O bound 任务优化
```
教训：I/O bound 任务可以增加 worker 数量超过 CPU 核心数

原理:
  - CPU bound: worker 数 = CPU 核心数 (避免上下文切换)
  - I/O bound: worker 数 = CPU 核心数 * 2-4 (等待 I/O 时可切换)

效果:
  - 2 workers → 4 workers
  - 扫描时间减少~15%
  - 无额外 CPU 开销
```
